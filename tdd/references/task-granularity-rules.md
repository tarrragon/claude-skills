# 任務粒度規則 — Use Case 驅動的自然拆分

> 來源：專案回顧分析。代理人每個 Ticket 執行 10-30 分鐘，表示拆分粒度不足。
> 目標：單一 Ticket 執行時間 **3-7 分鐘**，最長不超過 10 分鐘。

---

## 核心哲學

> **正確的粒度不是事後「拆」出來的，而是從 Use Case 設計時就自然「長」出來的。**

TDD 的意義：Use Case 定義單元 → 單元定義測試 → 測試定義實作。如果每一層的拆分都正確，Ticket 粒度自然合適。閾值檢查只是安全網，不是主要手段。

```
Use Case（業務行為）
    |
    v
單元識別（每個 Use Case 分解為獨立可測試的單元）
    |
    v
單元測試（每個單元 = 1 個測試 Ticket，含正常路徑 + 邊界條件）
    |
    v
整合測試（串連多個單元的端到端行為 = 獨立測試 Ticket）
    |
    v
實作（每個測試 Ticket 對應 1 個實作 Ticket = 1:1 映射）
```

**每一層的拆分都由上一層決定，而非由下一層的工作量反推。**

### 批量模式（新專案起手）

新專案從 doc 整批轉為 TDD 時，粒度單位從「行為單元」放大為「模組群組」：

```
Use Case（全部 UC 一次性展開）
    |
    v
模組群組（按 domain/package 分組）
    |
    v
群組測試 Ticket（每群組 7-15 個測試 = 1 個紅燈 Ticket）
    |
    v
群組實作 Ticket（每群組 1 個實作 Ticket，讓該群組測試全綠）
```

**與逐功能模式的差異**：

| 面向 | 逐功能（增量） | 批量（起手） |
|------|-------------|------------|
| 測試 Ticket 粒度 | 1 行為單元 = 1 Ticket | 1 模組群組（7-15 測試）= 1 Ticket |
| 實作 Ticket 映射 | 1:1（每測試一實作） | N:1（每模組群組一實作） |
| 適用場景 | 既有專案加功能 | 新專案從 doc 起手 |

**切換時機**：新專案第一輪紅燈→綠燈完成後，後續增量開發回到逐功能模式。

**認知負擔閾值仍適用**：批量 ticket > 15 個測試或 > 2 個概念時仍需拆分（見 `cognitive-load.md`）。案例：某專案首輪紅燈 ticket 含 35 個測試橫跨 4 個概念 → 拆為 3 個子 ticket（各 10-15 個測試、單一概念）。

### 實作 Ticket 拆分邊界判讀（測試變綠規則）

> **核心規則**：實作 Ticket 的拆分邊界必須對齊「測試從紅變綠」的驗收點。一個 Ticket 完成後若沒有任何測試狀態改變，它不應該是獨立 Ticket——它是另一個 Ticket 的步驟。

判讀三問：

| 問題 | 答「是」→ 不應獨立拆 | 答「否」→ 可獨立拆 |
|------|-------------------|-------------------|
| 這個 Ticket 完成後，有測試從 FAIL 變 PASS 嗎？ | 沒有（只定義 interface / 只建骨架） | 有（某組測試變綠） |
| 拆出的各部分能獨立跑測試嗎？ | 不能（共用 type 未定義會編譯失敗） | 能（各自 package 獨立編譯） |
| 不同部分共用同一組 type/error/constant 嗎？ | 是（Go internal package / Python 同模組） | 否（各自獨立模組） |

**反模式**：按 Spec FR 拆（輸入驅動），每個 FR 一個 Ticket。FR 是需求分類不是實作邊界——多個 FR 可能共用 type 定義、共用 error handling，分開實作會產生介面不一致。

**正確做法**：按「哪組測試變綠」拆（輸出驅動），每個 Ticket 完成後有一組測試從紅變綠。

**案例——拆分邊界未對齊測試變綠**：

某服務有 4 個模組（資料模型 / 儲存層 / 輸入端 / 查詢端），按 Spec FR 各拆 1 個實作 Ticket。用判讀三問檢查「資料模型」Ticket：

1. 完成後有測試從 FAIL 變 PASS？→ 沒有（只定義 interface，不實作邏輯）
2. 能獨立跑測試？→ 不能（儲存層和輸入端都引用模型 type）
3. 共用 type？→ 是（4 個模組共用同一組 struct）

三問全部指向「不應獨立拆」。按「測試變綠」邊界，應合併為 1 個 Ticket（全模組一次實作 → 全部測試變綠），或按「模型+儲存 變綠 / 輸入端 變綠 / 查詢端 變綠」拆——每個子 Ticket 完成後都有測試狀態變化。

## 第一機制：引導式拆分（Phase 0-1）

### 從 Use Case 到單元

設計 Use Case 時，問自己：

> 「這個 Use Case 包含哪些**可獨立驗證**的行為單元？」

每個行為單元應該是：
- 可以用一個 Given-When-Then 場景完整描述
- 可以獨立寫一個測試來驗證
- 失敗時不會影響其他單元的測試結果

**範例 — Chrome Storage adapter**：

原始 Use Case：「系統以 tag-based 結構儲存書籍分類資料」

正確拆分為行為單元：

| 行為單元 | Given-When-Then | 獨立可測？ |
|---------|----------------|-----------|
| 定義儲存結構 | Given tag schema → When 驗證結構 → Then 合法/拒絕 | 是 |
| 建立 tag | Given 有效 tag → When createTag → Then 儲存成功 | 是 |
| 讀取 tag | Given 已存在 tag → When getTag → Then 回傳資料 | 是 |
| 更新 tag | Given 已存在 tag → When updateTag → Then 資料更新 | 是 |
| 刪除 tag | Given 已存在 tag → When deleteTag → Then 資料移除 | 是 |
| 配額檢查 | Given 接近配額 → When 寫入 → Then 拒絕或警告 | 是 |

6 個行為單元 = 6 個測試 Ticket = 6 個實作 Ticket。每個 3-5 分鐘。

**對比原始做法**：1 個 Use Case → 1 個測試 Ticket（29 min）→ 1 個實作 Ticket（15 min）。

### 從單元到邊界條件

每個行為單元繼續問：

> 「這個單元在什麼情況下會走不同路徑？」

| 行為單元 | 正常路徑 | 邊界條件 |
|---------|---------|---------|
| 建立 tag | 有效資料 → 成功 | 重複名稱？超長名稱？空名稱？配額滿？ |
| 刪除 tag | 存在 → 刪除 | 不存在？有子分類？被書籍引用中？ |

邊界條件和正常路徑屬於**同一個行為單元**，放在同一個測試 Ticket 中。
不同行為單元的邊界條件**不要合併**到一個「邊界條件總 Ticket」。

### 從測試到實作的 1:1 映射

```
Phase 2 測試 Ticket          Phase 3b 實作 Ticket
──────────────────          ─────────────────────
T-001: 建立 tag 測試    -->  I-001: 建立 tag 實作
  - 正常建立                   - createTag 方法
  - 重複名稱拒絕               - 重複檢查邏輯
  - 空名稱拒絕                 - 驗證邏輯

T-002: 刪除 tag 測試    -->  I-002: 刪除 tag 實作
  - 正常刪除                   - deleteTag 方法
  - 不存在回傳錯誤             - 存在性檢查
  - 被引用時拒絕               - 引用檢查邏輯

T-003: 整合測試         -->  I-003: 整合修正（如有）
  - 建立後讀取一致             - 整合層連接
  - 刪除後查詢回 null
```

**禁止的模式**：

| 禁止 | 原因 | 正確做法 |
|------|------|---------|
| 「撰寫 X 的所有測試」 | 所有單元混在一個 Ticket | 每個行為單元一個測試 Ticket |
| 「實作整個 CRUD」 | 4 個獨立操作合成一個 | 每 1-2 個操作一個實作 Ticket |
| 「邊界條件測試」（獨立 Ticket） | 邊界從單元中抽離 | 邊界跟隨所屬單元 |
| 整合測試包含單元驗證 | 整合與單元混在一起 | 整合只驗串連，單元各自負責 |

## 第二機制：安全網閾值（派發前檢查）

> 引導式拆分是主要手段。以下閾值是安全網 — 如果超標，表示第一機制可能遺漏了。

| 指標 | 上限 | 超標意味著 |
|------|------|-----------|
| 驗收關注點數 | 2 個 | 可能混入多個行為單元 |
| 修改檔案數 | 2 個 | 可能跨越模組邊界（測試檔案不計入） |
| 關注點數 | 1 個 | 需要「和」或「+」描述 = 不是一個單元 |
| 預估執行時間 | 10 分鐘 | 行為單元可能還能再拆 |

> **驗收關注點 vs 驗收條件項目**：正常路徑 + 邊界條件屬於同一個行為單元 = 1 個關注點。「建立 tag 成功 + 重複名稱拒絕 + 空名稱拒絕」= 1 個關注點（建立 tag），不是 3 個。

**超標時的處理**：回到第一機制，重新檢視 Use Case 拆分是否充分，而非機械式按閾值切割。

## 反模式目錄

### 反模式 1：按優先級打包（Priority Bundling）

把 Phase 4 審查的多個發現按 P0/P1/P2 合成一個 Ticket。

> 詳細案例：cases/task-over-bundling-anti-pattern.md
>
> 根因：Phase 4 發現不是 Use Case 的行為單元，每個發現本身就是獨立的修復單元。
> 正確做法：每個發現 = 一個 Ticket，按優先級標記，可並行派發。

### 反模式 2：全範圍重寫（Full-Scope Rewrite）

「重寫 X 模組」作為一個 Ticket。

> 詳細案例：cases/rewrite-scope-split-strategy.md
>
> 根因：「重寫」不是一個行為單元，它是多個行為單元的總和。
> 正確做法：回到 Use Case，識別模組內的獨立行為單元，逐一建立 Ticket。

### 反模式 3：邊界條件獨立成 Ticket

把多個行為單元的邊界條件抽出來合成一個「邊界條件測試 Ticket」。

> 根因：邊界條件屬於行為單元，不是獨立關注點。把它抽離會讓測試 Ticket 和實作 Ticket 的對應關係斷裂。
> 正確做法：邊界條件跟隨所屬行為單元，在同一個測試 Ticket 中。

### 反模式 4：整合測試包含單元驗證

整合測試 Ticket 中同時驗證單元行為和串連行為。

> 根因：整合測試的目的是驗證「單元之間的串連」，不是重複驗證單元本身。
> 正確做法：整合測試只驗證端到端流程和單元間的資料傳遞，單元行為由各自的測試 Ticket 負責。

## 各 Phase 的應用

### Phase 1（規格設計）— 拆分的源頭

| 設計產出 | 應包含 |
|---------|--------|
| Use Case 描述 | 明確列出包含的行為單元 |
| 行為單元清單 | 每個單元有 Given-When-Then 場景 |
| 邊界條件清單 | 每個邊界歸屬到對應的行為單元 |
| 整合場景 | 哪些單元需要串連驗證 |

> **Phase 1 的品質決定了後續所有 Phase 的粒度。** 如果 Phase 1 只產出一個大 Use Case 而不分解行為單元，Phase 2/3 的 Ticket 粒度容易過大。

### Phase 2（測試設計）— 粒度驗證點

Phase 2 是驗證 Phase 1 拆分是否充分的最佳時機：

- 如果一個測試 Ticket 需要寫 > 5 個 test case → 行為單元可能還能拆
- 如果一個測試 Ticket 涵蓋多個不相關的 Given 條件 → 混入了多個行為單元
- 如果邊界條件無法歸屬到某個行為單元 → Phase 1 的單元識別不完整

### Phase 3b（實作）— 1:1 映射

每個測試 Ticket 對應一個實作 Ticket。不多、不少、不合併。

### Phase 4（重構/修復）— 每個發現獨立

Phase 4 不遵循 Use Case 拆分邏輯（因為它處理的是已完成實作的問題），但遵循「一個發現 = 一個 Ticket」原則。

## 派發前檢查清單（安全網）

- [ ] Ticket 對應到一個明確的行為單元？
- [ ] 驗收關注點 <= 2 個？（正常路徑 + 邊界條件 = 同一關注點）
- [ ] 預期修改檔案 <= 2 個？（測試檔案不計入）
- [ ] 標題不含「+」「和」「及」連接多個動作？
- [ ] 如果超標，已回到 Phase 1 重新檢視 Use Case 拆分？
