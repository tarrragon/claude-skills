# 冷讀執行代理人協定

這份給 golden-path-validation 階段 3 用:怎麼派、怎麼 prompt 冷讀執行代理人,讓他們像真實陌生人一樣照指引跑、回報有用的東西。

## 代理人的基本設定

- **fresh context**:用不帶你當前對話脈絡的新代理人。他們對這份指引 / repo 的唯一認識,就是指引本身——這才是真陌生人。
- **背景平行跑**:多個代理人同時在各自的乾淨環境跑、互不干擾(容器名各自帶後綴);主脈絡只收精煉報告。
- **涵蓋真實目標矩陣**:每個要支援的環境(發行版 / 架構 / 權限情境 / 有無圖形)各派一個執行型代理人,外加一個純理解冷讀(不執行)。

## Prompt 結構

每個執行型代理人的 prompt:

```text
你是一個剛發現 <repo/指引> 的陌生人,沒有任何預先知識。在一個全新的 <目標環境> 裡,
只照 <指引的哪一段> 走到 <目標終點>,並誠實回報陌生人照做會不會卡。

# 硬紀律
- 只照指引做,不要為了先搞懂而預讀底層腳本。只有指引本身讓你卡住、才去翻,
  並記下「指引沒講、得翻腳本才知道」這件事。
- 真的在乾淨的可拋棄環境(容器 / VM)執行,不是評論可讀性。
- 非互動:環境沒有 TTY,互動提示會 hang;每步加 timeout,hang 記為 finding。
- 從指引指定的來源取得內容(如從公開 repo clone),測「陌生人拿到的版本」。

# 回報四層
1. 理解:每步光看文字懂不懂要做什麼?哪一步的指令 / 前提 / 選擇不清楚、要猜?
2. 執行:每步真實結果。哪裡失敗 / hang / 報錯?錯誤訊息?你做了哪些指引沒寫的動作才過關?
3. 驗證:跑指引的 verify 步驟,完整輸出。它的 PASS/FAIL 跟你實際看到的環境狀態相符嗎?有無誤報?
4. 意想不到的狀況:任何指引沒預期到的事(缺工具、權限、互動提示、網路、架構相關…)。

# 收尾
測完清掉你的環境(容器 rm),不改 repo、不 push、不動宿主機。
報告結構化、控制篇幅。
```

純理解冷讀代理人:同樣 fresh、但**不執行**,只讀指引 + verify 腳本,回報「光讀懂不懂、每步跟不跟得上、哪些是只有作者懂的行話、前提有沒有交代、選擇點有沒有引導、危險動作有沒有標」。它抓的是理解層缺口,跟執行型抓的執行層缺口互補。

## 判讀回報

- **跨代理人 convergence**:多個代理人都中的同一個 finding 優先序最高——那是穩定的真缺陷,不是單一環境的偶發。
- **分清三種 finding**:①指引真 bug(硬編假設、缺步驟、寫錯)②環境 quirk(只在某容器 / 模擬出現、真機無關——要隔離、別當指引 bug)③理解缺口(讀不懂、要猜——補文字)。三種修法不同。
- **代理人「做了指引沒寫的動作才過關」是金礦**:那個動作就是指引缺的一步。
- **代理人回報「clean / 都過」也要對照 verify 實際輸出**:別只信結論,看它的執行痕跡。

## 邊界

- 冷讀代理人只證明「照指引在乾淨環境跑得起來」;跑不起來是硬證據,跑得起來仍要注意 verifier 誤報(見 principles/verify-the-verifier.md)。
- 代理人可能在陌生環境卡住繞不出來——prompt 要它「卡住就回報、別無限重試」,避免燒時間在單一環境的死路上。
