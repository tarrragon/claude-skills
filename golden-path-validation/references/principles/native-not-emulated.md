# 原生非模擬:架構敏感的驗證要在目標架構上跑

> **角色**:本卡是 `golden-path-validation` 的支撐型原則(principle)、被 SKILL.md 階段 4「環境保真度」引用。
>
> **何時讀**:要在跟目標不同的 CPU 架構上、或用 CPU 模擬跑驗證環境時(如在 arm64 主機上跑 amd64 image、或反之)。

架構敏感的驗證要在目標架構的原生環境跑;跑在 CPU 模擬(qemu 這類)下的環境會給假通過與假失敗。模擬層對多數應用邏輯透明,但對貼近核心的行為不透明——sandbox、seccomp、LSM(如 Landlock)、SIMD、特定 syscall 的支援與語意,在模擬層跟原生可能不同。所以「在模擬環境驗過了」對這類行為不算數。

## 模擬給的兩種假結果

- **假失敗**:一個在真機好好的操作,在模擬下因為某個 syscall 被模擬層擋掉 / 沒實作而報錯。你以為指引有 bug、去追一個不存在的問題。
- **假通過**:一個真機會出事的問題,在模擬下剛好沒觸發(模擬層的行為比真機寬鬆),你以為過了、發布後在真機才爆。

兩者都讓「這次驗證」失去意義:綠燈不能信、紅燈也不能信。判準是問一句:**這個行為依不依賴 CPU 架構或貼近核心的機制?** 依賴,就必須原生跑。

## 常見的踩法

- 手邊主機是 arm64、隨手 `docker run` 拉了一個只有 amd64 的 image → 靜默走模擬。很多 image 只發布單一架構、`docker` 會 warning 但照跑,容易沒注意。
- 解法是明確挑目標架構的原生 image / 環境(確認 image 的 `Architecture` 跟主機一致、或用目標架構的專屬 image),而不是接受隨手拉到的那個。

## 跟「隔離環境 quirk」的關係

模擬是環境保真度的一種破口,容器的安全 profile 差異是另一種:有些失敗只在容器出現、真機不受影響(如某些核心 LSM 在受限容器套不起來)。兩者的共同紀律是——**先確認你在驗的失敗 / 成功,是指引本身的,還是環境保真度破口造成的**。是後者就隔離掉(換原生環境、或放寬對應限制),不然會拿環境的假象當指引的判決。
