# 執行勝過審讀

> **角色**:本卡是 `golden-path-validation` 的支撐型原則(principle)、被 SKILL.md 的核心命題與階段 3 引用。
>
> **何時讀**:要判斷一份 follow-along 指引「到底成不成立」、猶豫「讀過一遍看起來對就好」還是「真的跑一次」時。

一份可執行的指引到底成不成立,審讀給不出答案、執行才給得出。文件層的 fact-check——每條指令語法對嗎、順序合理嗎、宣稱跟官方文件一致嗎——會通過,但那只證明「文字自洽」,不證明「照做跑得通」。真正卡住陌生人的,往往是文字層面完全正常、只有在真實機器上執行才會爆的東西。

## 審讀結構性看不到的三類

- **硬編假設**:腳本寫死某個前提(如 `sudo`、某路徑存在、某使用者身分),在作者的機器上成立、換個環境就 exit 非零。文字讀起來沒問題,因為那個假設沒被寫成一句話、而是藏在一個指令的形狀裡。
- **環境隨時間 / 跨平台漂移**:套件名被移除 / 改名、某工具在別的發行版沒打包、預設行為改版。指引寫的當下對,執行的當下已經不對。審讀對著「寫的當下」看,執行對著「現在」看。
- **驗證工具自己的 bug**:你寫來確認成功的檢查,本身可能讀錯層而誤報。這只有拿真實部署結果去跑它才會現形(見 verify-the-verifier)。

## 為什麼審讀天生漏掉這些

審讀者(作者、或跟作者共享脈絡的人)腦裡有太多沒寫進指引的前提,會不自覺地把缺口補上、讀起來就通了。執行沒有這種腦補:一個沒有前置知識的過程在乾淨機器上跑,缺什麼就當場缺什麼、假設不成立就當場報錯。這也是為什麼驗證要交給**冷讀 + 執行**,而不是再讀一遍。

## 操作結論

- 一份會被別人照做的指引,發布前至少讓它在乾淨環境真的跑一次;跨多環境就每個環境各跑一次。
- 「跑不起來」是硬證據、直接指出缺口;「跑得起來」是必要條件、還要配合 verifier(且 verifier 自己要先被驗)。
- 代理人「做了指引沒寫的動作才過關」——那個補上的動作,就是指引漏掉的一步,比任何審讀意見都精準。
