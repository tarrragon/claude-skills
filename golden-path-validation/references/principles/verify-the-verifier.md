# Verifier 也要被驗:你的驗證工具是一隻會讀錯層的眼睛

> **角色**:本卡是 `golden-path-validation` 的支撐型原則(principle)、被 SKILL.md 階段 2 與反覆陷阱引用。
>
> **何時讀**:寫了一支 verify 腳本 / 檢查來確認「有沒有部署成功 / 跑成功」、要開始信它的結果時。

你寫來確認成功的驗證工具,本身也是一隻會讀錯層的眼睛。診斷時「讀權威狀態、不靠肉眼猜」的紀律,對 verifier 一樣適用——一個 naive 的檢查可能查在錯的層、對一個健康的環境給假陰性,讓你以為部署壞了、去修一個不存在的問題。所以 verifier 不是寫完就能信,它要先被驗。

## 讀錯層的典型:符號連結與目錄摺疊

一個常見的假陰性:檢查「某個設定檔有沒有被 symlink 部署」時,直接判「這個 leaf 檔是不是 symlink」。但很多部署工具會**摺疊目錄**——當目標父目錄不存在時,建的是**目錄層級**的 symlink,裡面的 leaf 檔是穿透那層目錄 symlink 走到來源的實檔。對 leaf 檔判「是不是 symlink」→ false → 假陰性,即使部署完全正確。

權威讀法不是判單檔的符號連結屬性,而是比對「它到底解析到哪個檔」:`test <leaf> -ef <來源檔>`(比對 inode、跟隨符號連結,摺疊與否都對)或逐層 `ls -ld` 看哪一層是 symlink。差別是——`-L` 問「這個路徑本身是不是符號連結」(會被摺疊騙),`-ef` 問「這兩條路徑是不是同一個檔」(問到權威)。

## 怎麼驗 verifier

- **拿一個「已知正確」的環境跑它,確認它報 pass**。verifier 在你確定沒問題的環境上如果報 FAIL,那是 verifier 的 bug、不是環境的問題——先修 verifier。
- **假陰性比假陽性更危險**:verifier 誤報 FAIL,會讓一個好的部署被當成壞的、觸發不必要的「修復」;誤報 PASS 則讓壞的溜過。兩種都要防,但健康環境報 FAIL 是最該優先抓的。
- **verifier 的每個檢查都要能對應到權威狀態**:它查的是這件事真正的權威來源嗎(inode、程序的權威狀態、協議層狀態),還是一個容易被形態變化騙的表象(單檔 symlink 屬性、程序名字串、畫面元素)?

## 為什麼容易漏掉這一層

寫 verifier 的當下,注意力在「要檢查哪些項目」,很少停下來問「我這個檢查方式本身會不會誤判」。加上 verifier 通常在部署流程末端才第一次跑,它的假陰性很容易被誤讀成「部署那步壞了」,往上游追半天、其實是這隻眼睛讀錯層。把「verifier 也要被驗」當成部署驗證的固定一步,而不是等它誤報才想到。
