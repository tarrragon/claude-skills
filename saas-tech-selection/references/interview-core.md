# Interview Core：定錨 + 核心問題

本檔涵蓋兩個階段：定錨問題建立規模假設（訪談最開頭）、核心問題把需求轉成技術維度觸發訊號（在操作盤點與 domain / event 切分完成之後）。核心問題的輸入是操作盤點產出的操作清單、與 domain / event 切分產出的 domain map / event catalog — 問題錨在具體的操作與 event 上、不是抽象的「你的系統」。

訪談節奏：問題總量不設限、問漏才是成本；每輪 3-5 題讓使用者好消化、輪數跟著覆蓋率走。定錨階段允許開放問（請使用者描述產品）、核心問題用選擇題收斂、答不清楚的題目持續追問或轉成 tripwire。

---

## Stage 0：定錨問題

定錨的責任是建立「這個專案的預設方向」。同一個技術問題在不同定錨下答案不同：兩人團隊三個月上線的 B2C MVP、預設走 managed 服務與 PaaS；十人團隊有維運經驗的 B2B 產品、可以承擔自管元件換取成本與控制力。

| 問題     | 問法                                             | 答案如何影響後續                                                                    |
| -------- | ------------------------------------------------ | ----------------------------------------------------------------------------------- |
| 產品形態 | B2B、B2C、還是內部工具？                         | B2B 帶出合約與合規壓力、B2C 帶出流量不確定性、內部工具可以降低入口防護等級          |
| 動機     | 目標是營收、學習、還是解決自己的營運問題？       | 答案是決策記錄理由欄的合法錨點 — 「自建換學習」「託管換速度」的權重都要能回指這格   |
| 租戶模型 | 多租戶共用一套服務、還是每客戶獨立部署？         | 多租戶把「租戶隔離」升級成 state-storage 與 security 的第一級問題                   |
| 預期規模 | 首年用戶量級（百 / 千 / 萬 / 十萬+）？資料量級？ | 決定 capacity 維度是否展開、決定 DB / 部署的起點規格                                |
| 團隊能力 | 幾個人？有人背過 production on-call 嗎？         | 維運經驗低 → 預設 managed / PaaS、自管元件每加一個都要確認「誰半夜處理它」          |
| 投入     | 團隊每週可投入的工程時數？                       | 一週十小時跟全職的選型分母不同、各維度推薦的理由要能回指這個數字                    |
| 上線時程 | 多久要上線？上線後迭代節奏？                     | 時程緊 → 防護底線維持、選型偏保守（團隊熟悉 > 技術新穎）；迭代快 → CI gate 權重提高 |

定錨答案寫進決策記錄的專案定錨段、後續每個維度的推薦都要能回指到定錨（「因為團隊兩人且無維運經驗、推薦 managed PostgreSQL 而不是自架」）。

### 交付形態 gate：自建是否成立

定錨問完、先過一個 gate 再進後續訪談：**這個產品現在值得自建嗎**。自建的前提是差異化在軟體本身；差異化在商品、內容、社群或服務品質的業務、現成平台可能是成本上更誠實的起點 — 跳過這個判斷、就是用整套自建流程回答一個不該自建的需求。

gate 的第一個澄清問：**這個軟體是使用者要賣的產品、還是他經營業務的工具？**「做一個給健身教練用的課表系統」有兩種身分：要賣給眾多教練的產品 — 市場上的垂直 SaaS 是競爭對手、不是交付選項、走自建並繼續完整訪談；教練管理自己學員的工具 — 同一批垂直 SaaS（課表、預約、POS）正是該優先評估的託管候選。使用者的第一句描述通常分不出兩者、先問這題再進判讀表。

回答「都是」「先自用、之後想賣」時、判讀表的垂直 SaaS 列與自建列同時命中、用時間判準裁決 — 核心問題是**付費外部用戶離現在多遠**：

- 已有外部買家承諾、或近期有對外交付時程 → 按「要賣的產品」走完整訪談；自家使用記為首租戶（dogfooding）、不另算一條業務流程。
- 外部用戶還在想像階段、現在唯一的用戶是自己 → 託管先行；平台野心寫成升級自建 tripwire、觸發例：第一個付費外部用戶出現、或客製需求超出託管平台 API 能承載。

使用者聽完託管對照後仍選自建時、訪談照常繼續、不重複勸退；把選自建的動機（練手 / 控制權 / 長期成本）原樣記進決策記錄的專案定錨段、讓未來重評者能還原當時的權重。

| 需求描述                                                                 | 建議形態                                      |
| ------------------------------------------------------------------------ | --------------------------------------------- |
| 能用「型錄 + 結帳」「表單 + 通知」「文章 + 頁面」描述完                  | 全託管平台（Shopify / Wix / Google Sites 類） |
| 把現成服務串成流程（表單進試算表、定時寄報表、收件觸發動作）             | 辦公生態自動化（Google Apps Script 類）       |
| 行動 app / SPA、後端需求是認證 + 資料同步 + 推播                         | BaaS（Firebase / Supabase 類）                |
| 內容為主、要客製版型與外掛功能                                           | 半託管 CMS（WordPress 類）                    |
| 業務流程落在某個垂直行業 SaaS 的標準域（預約、課表、POS、訂位）          | 垂直 SaaS（行業專用的託管形態）               |
| 產品本身是軟體、且軟體是要賣的產品（SaaS 工具、API 服務、平台）          | 自建 — 繼續本 skill 的完整訪談                |
| 核心流程在任何現成平台都要大量 workaround 才能表達                       | 自建、或先重新檢視流程是否過度客製            |
| 服務只有自己 / 家人用、跑在自有主機或私有網路、無對外使用者              | 個人自架工具 — 自建但走極縮減流程（見下）     |
| 開源開發者工具 / CLI 工具 / 自架服務（自用起步、源碼公開讓別人也能自架） | 開源自架工具 — 自建但走開源自架流程（見下）   |

**個人自架工具的極縮減流程**：服務的唯一使用者是自己（或家人）、跑在自有主機或私有網路、沒有對外使用者時、自建成立但訪談走極縮減 — 跳過 domain / event 切分與多租戶、只展開 security、deployment-platform、secret 三條;state-storage 多半 N/A（無 source-of-truth 資料庫、配置可重建）。反向問從「資料外洩的代價」換成「**這台機器被攻陷的代價**」 — 因為閘道後面是一台真實主機的完整存取權。認證模型也不同：不是 web-auth（帳號 / SSO）、而是裝置綁定 + 共享密鑰（見 `dimensions/security.md` 身份認證段）;入口形態常是 outbound tunnel 而非公網 LB（見 `dimensions/deployment-platform.md` 候選類型段）。決策記錄換成個人工具縮減記錄：三條展開維度的選型 + tripwire「從單人變多人 → 回完整訪談、認證升級為帳號系統」。

**開源自架工具的中間流程**：源碼公開、接受 issue 和 PR、別人在自己的環境部署。和「個人自架工具」的差異是多了部署可移植性、文件品質、CI 的需求。和「自建 SaaS」的差異是沒有多租戶、沒有付費模型。訪談走中間路徑 — Stage 1 操作盤點做（機器角色是主要操作主體、見 `user-operations-bdd.md` 的開發者工具操作主體段）；Stage 1.5 畫面狀態矩陣可選（唯讀 dashboard ≤ 3 頁可跳過）；Stage 2 domain / event 切分做（多元件協作需要明確邊界）、多租戶不做；Stage 4 所有維度照常展開、security 認證預設可選（同區網信任）但加 tripwire「collector 暴露到公網或有跨網路存取 → 加 HTTP auth / API key」。決策記錄格式同完整流程、多一欄「可移植性」（別人能 `go build && ./collector` 就跑起來嗎）。

**開源替代品 check**：gate 判為「自建」後、在進 Stage 1 前追加一個產品級的「買 vs 建」check — 市面上是否有功能近似的開源方案？差異化在哪裡值得自建而非 fork 或直接用？和 Stage 2 的 commodity domain check 不同 — commodity check 是 domain 級（認證外包、搜尋外包）、替代品 check 是產品級（整個產品有沒有現成的開源替代品）。差異化不明確時、先用現成方案驗證需求、寫 tripwire「現成方案不夠用的具體場景 → 自建」。

**反向問**：請使用者用一句話描述核心流程、然後對照 — 這句話是不是某個現成平台的官方首頁文案？是 → 該平台進候選；一句話裡含現成平台表達不了的環節、才是自建訊號。適用邊界：目標本身是做該類平台的競品（要賣的產品就是這種平台）時、命中只代表「託管先行、先用該平台驗證流程」、不取消自建線 — 自建線依上方時間判準寫成 tripwire。

gate 逐條業務流程過、不是整個專案一次判：行銷頁、內部後台、核心產品可以各自落在不同形態（行銷頁託管 + 核心產品自建是常見組合）。訪談時把使用者描述的流程拆開逐條判、混合結論直接寫進決策記錄的交付形態欄。

結論是託管形態時、訪談走**縮減流程**：後續階段（操作盤點、domain / event 切分、核心問題、技術維度）跳過、決策記錄仍要產出、換成**託管縮減記錄**、涵蓋下列項目：

1. **平台選擇與理由**：需求對照平台標準域、回指定錨（團隊能力 / 時程 / 預算）。對照逐條過四問：核心業務流程哪幾條是平台原生支援？客製需求平台的 plugin / API 承載得了嗎？資料與身分的匯出能力如何？量級成長後計價曲線長怎樣？
2. **可遷出保險**（進場時最便宜、逐項確認、依平台類型轉譯 — 電商平台重點看金流可攜、垂直 SaaS 重點看客戶名單與歷史資料匯出）：自有網域（DNS 自己控制）、資料定期匯出（確認格式可重新匯入）、客戶 email 名單同步到自有系統、金流可攜性評估（訂閱扣款授權能否轉移）、會員密碼雜湊多半不可攜的重設預案、平台 UI 裡長出來的業務規則文件化。
3. **升級自建的 tripwire**：平台內 workaround 程式碼持續成長、平台月費 + 抽成跟自建成本曲線交叉、核心客製需求連續被平台擋下、API rate limit / 配額開始影響業務、合規或客戶要求超出平台控制面深度、平台政策變更直接衝擊營收。任一觸發 → 回來走本 skill 的完整訪談。
4. **防護底線總表（適用項）**：縮減流程跳過維度展開、防護底線在訪談中只經過總表這一次 — 適用項（帳號安全、operator MFA、資料匯出管控、自有網域）逐條標「已納入 / 延後 / 不適用」、不可沉默省略。

## Stage 3：核心問題

### 問題 1：需求類型

**問法**：把操作盤點的操作清單與 domain / event 切分的 event catalog 攤開、逐項歸類後端工作：狀態保存（帳號、訂單）、讀取加速、非同步處理（寄信、報表、webhook — event catalog 的條目多半落在這）、即時推送（聊天、通知）、診斷（之後怎麼查問題）、交付（怎麼部署）、可靠性驗證（怎麼確認沒壞）、安全邊界（誰能存取什麼）。

**為什麼問**：操作清單涵蓋了功能面、但這些工作類型裡至少有四類（診斷、交付、可靠性、安全）在操作清單裡看不到 — 它們是「提供這些操作」的隱含義務。逐類點名把隱形需求拉到檯面、同時驗證 event catalog 沒有漏掉操作清單裡的非同步交接。

**答案路由**：

| 訊號                                  | 觸發                                                                                       |
| ------------------------------------- | ------------------------------------------------------------------------------------------ |
| 正式狀態（帳號 / 訂單 / 合約 / 金流） | `dimensions/state-storage.md`                                                              |
| 重複讀取、昂貴計算、session           | `dimensions/cache.md`                                                                      |
| request 外的可靠工作                  | `dimensions/async-queue.md`                                                                |
| 即時推送                              | `dimensions/async-queue.md`（pub/sub 段）+ `dimensions/deployment-platform.md`（長連線段） |

**反向追問**：「有沒有哪個動作、失敗了使用者不會立刻發現？」— 寄信失敗、webhook 漏發、報表算錯都屬於這類、它們是非同步維度的隱形觸發訊號。

### 問題 2：流量形狀

**問法**：流量是穩定成長、有可預期高峰（活動 / 開賣 / 月底結算）、還是完全未知？有沒有少數熱門資源吃掉多數流量的可能（hot key）？

**為什麼問**：流量形狀決定容量策略的型態 — 穩定流量靠容量規劃、可預期高峰靠事前準備、未知流量靠彈性與降級。MVP 階段多數答案是「未知」、這個答案本身就是有效資訊：它把 capacity 維度轉成 tripwire（「首次高峰活動前重評」）而不是現在展開。

**答案路由**：可預期高峰或首年破十萬用戶 → `dimensions/capacity-performance.md`；其餘記入 tripwire 總表。

**反向追問**：「如果明天被大量轉發、流量十倍、你預期系統先壞在哪？」— 答不出來是正常的、這題的目的是把「先壞在哪」寫進觀測與 tripwire。

### 問題 3：資料生命週期

**問法**：哪些資料要長期存在？哪些可以重建（快取、索引、報表）？有沒有資料受法規或合約要求保留或刪除（發票、個資、稽核紀錄）？

**為什麼問**：「可以重建的資料」與「不可重建的資料」走不同儲存策略 — 前者的備份壓力低、後者的備份與還原驗證是底線。保留 / 刪除要求直接觸發 security 維度的資料治理段。

**答案路由**：不可重建資料 → `dimensions/state-storage.md` 備份底線段；法規保留 / 刪除 → `dimensions/security.md` 資料保護段。

**反向追問**：「使用者要求刪除帳號時、哪些地方會殘留他的資料？」— 備份、log、第三方服務都是常見殘留點、SaaS 多數沒在 day one 想過這題。

### 問題 4：失敗代價

**問法**：延遲、重複、遺失、暫時不一致、停機 — 這五種失敗各自對產品造成什麼後果？哪一種是不可接受的？

**為什麼問**：失敗代價是選型的真正分水嶺。「訂單重複扣款不可接受」推導出 idempotency 底線；「停機一小時可接受」讓部署平台可以選簡單方案。沒有這題、所有維度的選型都會預設「全部都重要」、成本爆炸。

**答案路由**：重複 / 遺失不可接受 → `dimensions/async-queue.md` idempotency 段 + `dimensions/state-storage.md` transaction 段；停機代價高 → `dimensions/deployment-platform.md` 回滾段 + `dimensions/reliability.md`。

**反向追問**：「金流相關的操作、第三方（付款服務）timeout 時你的系統算成功還是失敗？」— 這題暴露分散式邊界的失敗語意、是 SaaS 最常見的事故源之一。

### 問題 5：成本模型

**問法**：目前最貴的資源是什麼 — 雲端費用、人力時間、事故風險、還是上市速度？每月基礎設施預算量級？

**為什麼問**：成本模型決定 managed vs 自管的全局傾向。人力最貴 → 全 managed；雲端費用敏感 → 接受自管換成本、但要把維運人力成本攤開講。這題的答案會被每個維度引用。

這裡的 managed vs 自管不是二元、而是一條外包深度軸（managed 基礎設施 / feature SaaS / 跨能力 bundle）、見 `principles/capability-outsourcing-depth.md`。兩個推論影響後面每個維度的展開：① 交付形態 gate 判「自建」的流程、不等於每個維度都自己跑機器 —— 自建流程內每個技術維度仍各自有 day-one 託管預設、每個 commodity domain 仍可被買掉；gate 的「託管 = 跳過」是流程級、維度級與領域級的外包是另一層、不被 gate 一次解決。② 一個跨能力 bundle 可以一次覆蓋多個維度（資料庫 + 認證 + 物件儲存 + 即時推送）—— 維度展開時若候選是這種 bundle、不要逐維度當獨立服務各選一次、改逐維度判斷「這塊用 bundle 值不值得換取整包遷出代價」、把結論收在同一筆 bundle 決策裡。

**答案路由**：寫進決策記錄的專案定錨段、作為各維度推薦的依據。

**反向追問**：「省下來的雲端費用、值多少個半夜被 page 的晚上？」— 自管選項的隱形成本要在這裡攤開。

### 問題 6：定位與備援

**問法**：錯誤發生時、你打算怎麼知道「哪裡壞了」？依賴的服務（DB、第三方 API）掛掉時、產品要降級還是直接停？

**為什麼問**：這是「使用者沒想到的東西」密度最高的一題。功能開發者預設一切正常運作、診斷與降級是事故發生時才會後悔的投資。這題把 observability 與 reliability 兩個必展開維度的訪談鋪好路。

**答案路由**：`dimensions/observability.md`（必展開）+ `dimensions/reliability.md` 第三方依賴段。

**反向追問**：「凌晨三點服務掛了、誰會知道、多久會知道、第一步看哪裡？」— 三個子問分別對應 alert 路由、偵測延遲、debug 入口。

### 問題 7：安全邊界

**問法**：誰能存取哪些資料？管理者、客服、工程師各自看得到什麼？資料外洩的最壞情境是什麼？

**為什麼問**：安全是底線密度最高的維度、且事後補建的成本遠高於 day one 內建（權限模型重構、audit 補登都是大工程）。多租戶 SaaS 在這題還要確認租戶隔離的實作層級。

**答案路由**：`dimensions/security.md`（必展開）；多租戶 → 同檔租戶隔離段 + `dimensions/state-storage.md` 租戶資料模型段。

**反向追問**：「工程師能不能直接連 production DB？能的話、誰知道他連過？」— audit 缺口在這題現形、答案多半是「能、而且沒人知道」。

## 收斂判準

核心問題全部問完後、確認下列事項再進維度展開：

1. 每個必展開維度都拿到至少一個來自前面訪談階段（操作盤點 / domain / event 切分 / 核心問題）的具體輸入：操作、event 或核心問題的答案、不是憑空展開。
2. 觸發展開的維度有明確訊號依據；未觸發的維度在決策記錄標「未展開 + 觸發條件」。
3. 反向追問中答不出來的題目、轉成觀測項或 tripwire、不強迫使用者當場給答案。
