# User Operations（BDD 操作盤點）

本檔是操作盤點階段（訪談第一站、緊接定錨之後）的操作手冊。功能需求的唯一來源是「我們要提供使用者哪些操作」— 系統設計是這些操作的後果、不是起點。本階段產出操作清單：每個操作帶行為情境（Given / When / Then）、誤操作風險、引導設計、前端到後端的防護對應。這份清單同時是三個下游的輸入：domain / event 切分從操作推導 command 與 event、技術維度展開把防護錨在具體操作上、未來的測試直接以行為情境為骨架。

本階段的訪談姿態：**寫不出行為情境的操作、就是還沒想清楚的需求**。在這裡多問十題、比開發到一半發現「原來使用者還會這樣操作」便宜一百倍。追問到每個操作四欄齊備（情境 / 風險 / 引導 / 防護）為止、不因為問題多而收手。

## 盤點流程

### 第一步：操作主體枚舉

| 主體類型     | 常見成員                               | 最常漏的原因                             |
| ------------ | -------------------------------------- | ---------------------------------------- |
| 終端使用者   | 一般會員、付費會員、團隊成員           | 通常不會漏、但會漏「未登入訪客能做什麼」 |
| 組織內角色   | 團隊管理者、帳務負責人、被邀請未註冊者 | 邀請 / 移除 / 轉移擁有權的流程最常被跳過 |
| 營運內部角色 | 客服、營運後台、工程師                 | 「客服怎麼幫使用者解決問題」沒人當功能想 |
| 機器角色     | 第三方 webhook、排程任務、API client   | 機器也是操作主體、它們的失敗也是操作風險 |

#### 開發者工具 / 開源工具的操作主體

開發者工具的操作主體結構和 SaaS 產品不同 — 主要操作者是機器角色和開發者自己、沒有「終端使用者」和「組織角色」的分層。

| 主體類型     | 常見成員                                   | 說明                                                       |
| ------------ | ------------------------------------------ | ---------------------------------------------------------- |
| 開發者 owner | 部署者 + 設定者 + 資料消費者（三角色合一） | 同一個人做部署、設定和使用，操作面比 SaaS admin 寬但人數少 |
| 機器角色     | SDK 實例、daemon、hook、cron job           | 是主要操作主體，佔操作清單的大部分，失敗處理是設計核心     |
| 外部貢獻者   | PR 提交者、issue 回報者                    | 不操作 production、但需要能 clone → build → test → 本地跑  |

機器角色在開發者工具中的佔比通常超過 50%（SaaS 產品中機器角色通常是補充）。機器操作的行為情境寫法和人類操作相同（Given / When / Then），但失敗情境更重要 — 機器操作的失敗不會有人即時看到、必須有明確的降級或重試策略。

### 第二步：每主體操作枚舉

用「這個角色打開產品想完成什麼」枚舉、並沿生命週期問完整：進來（註冊、被邀請）、日常使用、狀態變化（升級、降級、權限變更）、離開（取消訂閱、刪除帳號、資料匯出）。生命週期尾端（離開類操作）是最常見的設計缺口 — 沒人想做、但合規與商譽都掛在上面。

### 第三步：每操作寫行為情境

每個操作至少一條主情境 + 一條失敗情境：

```gherkin
# 主情境
Given 已登入的團隊管理者、團隊有 3 名成員
When 他移除成員 B
Then 成員 B 立即失去存取、收到通知信、團隊席次釋出一席

# 失敗情境
Given 已登入的團隊管理者
When 他嘗試移除自己（唯一的管理者）
Then 操作被拒、提示「先指派另一位管理者」
```

情境寫法的規則：

1. **寫使用者可觀察的行為、不寫系統內部動作**：「訂單狀態變為已成立、且使用者收到確認信」合格；「寫入 orders 表、發 OrderPlaced 事件」是 domain / event 切分階段的產物、不進情境。
2. **Then 要可驗證**：每條情境就是未來的一條驗收測試；驗證不了的 Then（「系統處理該請求」）等於沒寫。
3. **失敗情境是一級公民**：權限不足、輸入錯誤、依賴故障、並發衝突 — 失敗情境的數量通常該多於主情境。

## 誤操作風險盤點

每個操作過一次風險維度表。風險的處理一律**前端引導與後端防護成對出現**：前端的引導防的是誤會、後端的防護防的是繞過 — 只有前端是裸奔、只有後端是難用。

| 風險類型   | 典型情境                     | 引導設計（前端）                       | 防護（後端）                                    |
| ---------- | ---------------------------- | -------------------------------------- | ----------------------------------------------- |
| 重複提交   | 雙擊送出、付款頁重整重送     | 按鈕送出後 disable、導向結果頁         | idempotency key、unique 約束                    |
| 中途離開   | 多步驟流程斷在第三步         | 草稿自動保存、進度指示、回來可續       | 狀態機允許 resume、半成品有過期清理             |
| 不可逆操作 | 刪除專案、取消訂閱、移除成員 | 二次確認、輸入名稱複誦、冷靜期         | 軟刪除 + 恢復窗口、操作進 audit log             |
| 批次操作   | 全選刪除、CSV 匯入、批次改價 | 預覽影響範圍、顯示筆數、分步確認       | dry-run 模式、單次上限、可回滾                  |
| 權限外嘗試 | 改 URL 的 id、呼叫隱藏 API   | 前端隱藏入口只是 UX、不當防護宣傳      | 物件層級授權檢查（每查詢驗 ownership）          |
| 輸入錯誤   | 格式錯、超長、貼上惡意內容   | 即時驗證提示、範例 placeholder         | schema validation、欄位白名單、長度上限         |
| 並發衝突   | 兩人同時編輯同一筆資料       | 顯示「他人正在編輯」、衝突時提示差異   | 樂觀鎖（version 欄位）或最後寫入策略 + 變更紀錄 |
| 誤解後果   | 「封存」以為是「刪除」       | 動詞精確、後果寫在按鈕旁、不靠 tooltip | 操作可逆性與字面承諾一致                        |

風險盤點的判讀原則：**風險等級跟著「不可逆程度 × 影響範圍」走**。可逆的單筆操作（改暱稱）給最輕引導；不可逆的批次操作（刪除整個工作區）給全套（複誦 + 冷靜期 + 軟刪除 + audit）。每個操作在表上對號入座、避免「全部都加二次確認」的均值化 — 到處都是確認框等於沒有確認框。

#### 機器操作的風險對應格式

機器角色（SDK 實例、daemon、webhook receiver）的操作沒有「前端引導」— 它們的風險對應用「API 契約」和「失敗處理策略」取代：

| 風險維度     | 對應設計                                                                            |
| ------------ | ----------------------------------------------------------------------------------- |
| API 契約     | 成功和失敗的 response 格式明確定義（HTTP status + body 結構），SDK 能據此判斷下一步 |
| 失敗處理策略 | 靜默降級（init 失敗不影響 app）/ retry with backoff / 報錯終止 — 每個操作明確選一種 |
| 離線降級     | 網路不可達時的行為（buffer 到本地 / 丟棄 / 阻塞等待）+ 恢復後的補發和去重機制       |
| 容量保護     | buffer 上限（記憶體 + 檔案）、FIFO 丟棄策略、滿時的 warning log                     |

**開發者工具的常見失敗處理模式**（供操作盤點參考）：

| 失敗場景           | 業界常見做法                                                           |
| ------------------ | ---------------------------------------------------------------------- |
| SDK init 失敗      | 靜默降級、後續 API 呼叫變 no-op（Sentry / Crashlytics / Datadog 皆同） |
| 收集端不可達       | 事件進離線 buffer、恢復後補發、以 event_id 去重                        |
| Schema 驗證失敗    | 拒絕 + 回 400（核心欄位）、接受但不驗結構（自由欄位）                  |
| Buffer 滿          | FIFO 丟棄最舊事件 + log warning                                        |
| Flush-on-exit 失敗 | best-effort、不阻塞 app 關閉                                           |

#### 保留策略的驅動模式

資料保留策略在 SaaS 產品和自架工具中的驅動因素不同：

| 驅動模式 | 約束來源               | 典型做法                                          |
| -------- | ---------------------- | ------------------------------------------------- |
| 時間驅動 | 商業方案計費、合規要求 | 保留 N 天後自動刪除（30 / 90 / 365 天）           |
| 容量驅動 | 自架工具的物理磁碟     | 磁碟使用率 > 閾值時清最舊或最低優先類型的資料     |
| 類型分層 | 不同類型資料的查詢價值 | error 保留久、event 保留短、metric 聚合後只留摘要 |

自架工具通常用容量驅動 + 類型分層的組合。保留期的根本問法是「你需要跑哪些查詢、每個查詢需要回溯多遠、回溯時需要逐筆原始還是聚合摘要」— 答案推導出分層保留 + 降採樣策略：逐筆原始保留短（天級、debug 用完即丟）、小時摘要保留中（季級、趨勢分析）、天/月摘要保留長（年級、留存和分群）。這個問法同步用在 `dimensions/state-storage.md` 的訪談問題中。

## 反向問

每個操作群盤完、用反向問挖沒想到的面：

1. 「使用者做完這個操作**馬上後悔**、產品給他什麼路？」— undo、恢復窗口、聯絡客服？答案是「沒有路」的操作要重看引導等級。
2. 「**兩個人同時**對同一筆資料做這個操作、會發生什麼？」— 並發是使用者層面的問題（誰的編輯活下來）、不只是技術問題。
3. 「使用者**最可能誤解**這個操作會做什麼？」— 動詞與後果的落差（封存 vs 刪除、離開團隊 vs 解散團隊）。
4. 「**客服接到**關於這個操作的求助電話、他需要看到什麼、能替使用者做什麼？」— 客服操作是隱藏的功能需求、現在不盤、上線後變成工程師手動改 DB。
5. 「這個操作**被自動化濫用**（腳本、爬蟲）會怎樣？」— 註冊、邀請、匯出是濫用熱區、防護需求從這裡進入 security 維度。

## 畫面狀態矩陣展開（Stage 1.5）

操作清單完成後、進入 domain/event 切分之前，把操作清單的「前端引導」欄展開成完整的畫面狀態矩陣。展開的目的是補上「前端引導」欄只描述顯示、容易遺漏操作和退出路徑的缺口。

展開步驟：從操作清單抽出涉及的畫面 → 每個畫面列出所有狀態 → 每個狀態填四欄（顯示 / 可用操作 / 進入條件 / 退出路徑）。**退出路徑欄為空 = UX 死胡同**，代表使用者進入後無法自行離開。

每個 gate（認證、網路、權限、環境）在矩陣中用三問展開：成功時做什麼、失敗時做什麼、使用者不知道發生什麼時做什麼。Gate 的「失敗」欄為空 = 使用者被擋住無法繼續。

操作清單產出只要有 UI 元件（不限於 mobile app — SPA、desktop app 都適用），就做畫面狀態矩陣展開。純 API 服務不需要。

**跳過判準**：畫面 ≤ 3 個且全部唯讀（沒有 gate、沒有使用者觸發的狀態轉換、退出路徑全是「關閉瀏覽器」）→ 確認無 gate 即可跳過矩陣展開。開發者工具的唯讀 dashboard 通常命中此判準。

完整的展開原則與步驟見 `principles/screen-state-matrix-expansion.md`。

## 產出格式

操作清單表、每列七欄：

| 操作 | 角色 | 主情境 | 失敗情境 | 風險類型 | 前端引導 | 後端防護 |
| ---- | ---- | ------ | -------- | -------- | -------- | -------- |

收斂判準（全部通過才進 domain / event 切分）：

1. 每個操作主體（含內部與機器角色）至少盤過一輪生命週期。
2. 每個操作四欄齊備：行為情境、風險、引導、防護。
3. 寫不出 Given / When / Then 的操作、已經追問到寫得出來、或明確標記「需求未定、範圍外」並記錄原因。
4. 反向問都問過、答案進表或進風險欄。
