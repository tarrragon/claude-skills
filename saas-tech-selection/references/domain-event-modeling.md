# Domain / Event Modeling（DDD 切分：SRP + OCP）

本檔是 domain / event 切分階段（緊接操作盤點之後）的操作手冊。責任是把操作盤點產出的操作清單轉成兩個 artifact：**domain map**（領域怎麼切、每個領域對外公開什麼）與 **event catalog**（什麼事實在領域之間流動）。技術選型掛在這個骨架下：transaction 邊界跟著 domain 走、queue 的訊息就是 event catalog 的條目、授權模型沿 domain 邊界配置 — 骨架沒立好、後面的技術維度都在對空氣選型。

本階段只用兩個原則：**domain 與 event 依 SRP 切分、domain 依 OCP 區分公開面與內部面**。SOLID 其餘三個原則（LSP / ISP / DIP）留給實作階段 — 初始化階段還沒有 class 階層、沒有 interface 簇、沒有依賴注入結構、提前套用會把訪談拖進實作細節。

---

## 推導鏈：operation → command → domain → event

每個使用者操作沿同一條鏈推導：

1. **Operation**（操作盤點的產物）：使用者可觀察的行為 —「團隊管理者移除成員」。
2. **Command**：系統被要求做的事 — `RemoveMember`。命令式、現在式。
3. **Domain**：唯一一個負責處理這個 command 的領域 — Team domain。一個 command 找不到唯一歸屬、或一個 domain 接到性質完全不同的 command 群、都是切分要重看的訊號。
4. **Event**：處理完成後對外宣告的事實 — `MemberRemoved`。過去式、不可否認、別的 domain 自己決定要不要反應（通知 domain 訂閱它寄信、計費 domain 訂閱它釋出席次）。

推導方向是固定的：**從操作出發、不從資料表出發**。從資料表出發會切出 CRUD 式的偽領域（UserManager、OrderManager — 它們的「責任」是一張表、不是一段業務）；從操作出發才會浮現業務領域（報名、計費、出貨、通知）、以及領域之間真正需要交換的事實。

## Domain 切分的 SRP 判準

SRP 在領域層的意思：**一個 domain 只有一個變更理由**。三個判讀問題：

1. **「這兩個概念會因為同一個業務原因一起改嗎？」** 定價規則改、會連動庫存邏輯嗎？會 → 考慮同 domain；只是資料有外鍵關聯 → 不構成同 domain 的理由。資料相關 ≠ 責任相同。
2. **「這個規則改變時、誰說了算？」** 決定權屬於不同的人（定價歸商務、庫存歸營運、出貨歸物流）→ 不同 domain。組織的決策邊界是領域邊界最誠實的線索。一人專案的決定權全部同一人、這題退化 — 改問「未來第一個協作者或外包進來、會先交出哪一塊？」、交接邊界即決定權邊界。
3. **「同一個詞在不同情境意思一樣嗎？」** 「商品」在型錄情境是描述與圖片、在訂單情境是價格快照與數量、在庫存情境是儲位與數量 — 同名不同義就拆開、各 domain 持有自己需要的版本、用 id 關聯。把三種「商品」塞進一個 model 是日後所有耦合的起點。

### 切太粗與切太細的訊號

| 訊號                                                           | 判讀                         |
| -------------------------------------------------------------- | ---------------------------- |
| 一個 domain 的改動頻繁波及看似無關的功能                       | 切太粗、藏著多個變更理由     |
| 操作盤點的行為情境一條跨越多個不相干概念才寫得完               | 切太粗                       |
| 兩個 domain 永遠在同一個 PR 裡一起改                           | 切太細、兩者共用一個變更理由 |
| 兩個 domain 之間的 event 來回打乒乓（A 發 B 收、B 馬上發回 A） | 切太細、或邊界畫錯位置       |

MVP 階段的 domain 數量參考：3-6 個是常態（帳號 / 核心業務一到兩個 / 計費 / 通知）。超過 8 個先懷疑切太細、只有 1-2 個先懷疑用資料表思維在切。

## Commodity domain check：這塊 domain 該建還是該買掉

切完 domain map、定 event 之前、逐 domain 過一道買 vs 建的檢查：**這個 domain 是不是現成 feature SaaS 已經做完的標準能力？** 命中的 domain 不該被當成要自建的領域一路模 event、展開技術維度 —— 它該變成一條整合邊界（provider adapter + webhook）、整塊移出 build scope。這是領域層的外包、槓桿比技術維度層大：買掉一個 domain 等於連它要展開的 state-storage / security / async-queue 一起省掉。深度判讀見 `principles/capability-outsourcing-depth.md`。

逐 domain 對照常見的 commodity 清單：

| Commodity domain       | 現成 feature SaaS（例）                            | 命中後的處理                                                                                                                                                                          |
| ---------------------- | -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 認證 / 身分            | 託管身份服務（Auth0 / Cognito / Firebase Auth 類） | 已在 `dimensions/security.md` 的 day-one 預設處理、commodity check 命中認證時指回該維度、不在 domain map 模內部 auth 流程                                                             |
| 金流 / 訂閱計費        | Stripe / 各地金流服務                              | 付款 domain 變整合邊界：你的 domain 只持有「訂單與付款狀態的對應」、扣款、卡片、發票、PCI 合規全在 vendor（但 KYC / AML / 資金存管等業態合規不隨接 vendor 轉移）；接 webhook 同步狀態 |
| 表單 / 問卷 / 資料蒐集 | 表單平台（Typeform / SurveyCake / Ragic 類）       | 不模表單 domain、改為「表單結果如何流回自己的 domain」的整合題；常是非工程團隊的主力選擇                                                                                              |
| 站內搜尋               | 搜尋服務（Algolia 類）                             | 不自建搜尋 domain、改為「哪些資料 push 到搜尋服務、查詢怎麼接」                                                                                                                       |
| 通知 / Email / 簡訊    | 投遞服務（SendGrid / Twilio 類）                   | 通知 domain 退化成「訂閱領域 event → 呼叫投遞服務」的薄層、deliverability 全在 vendor                                                                                                 |
| 物件儲存 / 檔案 / CDN  | 物件儲存 + CDN（S3 / R2 / Cloudinary 類）          | 不自建檔案儲存與分發、改為「哪些檔案放它、存取控制與簽名 URL 怎麼接回 domain」                                                                                                        |
| 後台 / internal CRUD   | internal tooling（Retool / 平台內建 admin 類）     | 「完整後台」常被當自建理由、但 admin CRUD 是 commodity、買掉把工程時間留給真正客製的業務流程                                                                                          |

命中與否的判準跟 domain SRP 一致、但問的是相反方向：**這塊是不是產品的差異化？** 是差異化核心 → 自建（再貴也建）；只是「每個同類產品都要、做得再好也不加分」的標準能力 → 預設買掉。一個整合層型的產品（核心只有一兩個自建 domain、其餘全接現成服務）是合理結構、不是偷懶。

接縫提醒：買掉多個 commodity domain 會在它們之間長出整合接縫 —— 同一份使用者資料可能同時在自己的 DB 跟外部表單 / 金流服務裡、一致性與同步變成新的維護面。每買掉一個 domain、在決策記錄記一條「這道接縫由誰維護、資料一致性怎麼保證」。

買掉的 domain 在 domain map 上仍佔一列、但標注「外包（vendor）+ 整合邊界」、公開面寫「與 vendor 的整合介面（webhook / API）」、內部面寫「無（vendor 持有）」—— 讓後續維度展開知道這塊不需要 state-storage / security 的完整訪談、只需要 reliability 維度的第三方依賴訪談（sandbox / 失敗語意 / 費用 alert / 驗簽）。例外：金流這類受業態合規約束的能力、命中後仍要過一次業態合規檢查（KYC / AML / 資金存管 / 交易留存不隨接 vendor 轉移）、不能只做 reliability 訪談。

### 命中後逐能力該問什麼（買 vs 建判讀問題）

commodity 清單裡每塊能力的「買 vs 建」判準形狀不同 —— 同樣是現成 feature SaaS、認證撞的牆跟搜尋撞的牆不一樣。命中某塊後、用該能力自己的問題追、不要套同一張通用表。下面每塊的問題集來自這類能力的真實買 vs 建決策反覆出現的因子（成本曲線、遷出鎖定、合規、客製天花板、整合接縫）；數字依各 vendor 當下定價自查、不在此固化。

有一條判準橫切所有 feature SaaS、值得在逐能力問之前先立起來。commodity 能力的需求大多跟同業共通、真正區分產品的往往只剩一小塊（粗估八成共通、兩成差異化、比例只是示意）。第一道要問的其實是「那塊差異化到底存不存在」—— 有些 commodity 整塊就是買、根本沒有要守的那一塊：一個只負責把通知寄出去的能力、不會因為寄得比同業好而贏、這條判準對它直接跳過。確認有那塊差異化、才問第二層：vendor 願不願意讓你程式化改它的預設行為、把那塊長出來。帶訪談對象問這條時、別停在「它有沒有 API」這麼淺、而是逐塊對著差異化需求驗 —— 搜尋撞的是相關性排序能不能覆寫、表單撞的是結果資料能不能用 API 取回、後台撞的是能不能掛自訂 workflow。每塊的答案都回到同一個判斷：vendor 留給的擴展空間、夠不夠把那塊差異化長出來。這條軸不取代下面各能力的問題集、它是問每塊問題前先校準的尺。

**認證 / 身分**：

1. 這家的 password hash 匯出政策是什麼？（不可攜帶 = 之後要換得全員強制改密碼 = day-one 客服爆量的 UX 事故；認證最容易被忽略卻最硬的鎖定點、選型當下就要問、不是上線後才發現）
2. 有沒有任何客戶或法規要求 air-gap / on-prem / 資料住所？（離線、無對外呼叫的部署需求直接讓雲端託管 SaaS 出局）
3. 預期成長下 MAU / per-seat 定價的三年曲線長怎樣？有沒有漲價條款？（中量級後陡升、vendor 主動漲價是最常見的被迫遷移觸發器）
4. roadmap 上多久會需要企業 SSO / SAML / SCIM？幾個企業客戶？（這是 gating 企業營收的成本中心、自建成本高、是「買」的最強理由）
5. auth 的資料模型是不是產品差異化的一部分？（是 → 自建保留演進空間；否 → 預設買）
6. 既有使用者怎麼導入？（從現有系統或別家 IdP 遷入的 import 路徑 —— 跟 hash 匯出政策對稱的入口側、轉進來時的摩擦同樣常被忽略）

**金流 / 訂閱計費**：

1. 預期交易量 × 手續費% 到不到「自建或 orchestration 才划算」的拐點？
2. PCI-DSS 認證與長期維護（卡組織規則、法規變動）誰扛？
3. 需要幾個在地支付方式 / 幣別？（跨國、redirect / async 流是大型平台自建的核心驅動）
4. 要不要 multi-PSP 路由容錯（orchestration layer）？單一 PSP 的可用性風險可接受嗎？
5. 帳務正確性（double-entry、idempotency、避免重複扣款）誰保證？
6. 金流特有提醒：這塊常不是全有全無、而是「規模後部分自建」—— 先用單一 PSP、量大後在其上疊自建 orchestration / 在核心市場 direct acquiring。問「現在買、之後哪一層會想拿回來」。

**站內搜尋**：

1. 相關性調校是不是長尾痛點？（自建最大的隱性成本是 match 掉開箱即用的相關性，垂直領域 / 特殊斷詞尤甚）
2. 預期記錄量級在不在「規模拐點」之上？（量大到一定程度後自建開始便宜、買的成本曲線只在小規模佔優）
3. 計費是 usage-based（記錄數 × 查詢 / 操作次數）還是 cluster / infra-based（規格固定）？流量尖峰下前者可預測嗎？
4. 資料要不要留在自家 VPC（資料主權 / 合規）？
5. 搜尋服務有沒有相容的前端搜尋 adapter（讓 UI 元件不必重寫）、能不能降低之後的遷移摩擦？（遷移摩擦本身是 buy 決策的變數）
6. 重建索引（reindex）多痛、資料更新到搜尋可見的新鮮度延遲可接受嗎？（schema 變動或全量重灌的代價、自建與託管都要問的營運面）

**表單 / 問卷 / 資料蒐集**：

1. 用量 / response 數計費的上限會不會撞？feature gating（條件邏輯、多步）卡在哪一檔？
2. 客製天花板夠不夠？（品牌 rendering、多步 wizard、條件分支、side-by-side layout）
3. **資料怎麼接回來？** —— 這是表單的核心接縫：webhook payload 是 multipart 還是 JSON？要不要二次 call API 拿完整 / 加密答案再解密？（接縫的工作量常被低估）
4. 資料品質怎麼保？（無 OTP / email 驗證 → 假填答）
5. 誰維護：非工程師填表就能改（no-code 端）、還是需要 API 級控制（dev-tool 端）？這決定選光譜哪一端。

**後台 / internal CRUD**：

1. per-seat / per-viewer 計費隨團隊與使用者成長的年費？會不會造成「限制誰能用」的反向誘因？
2. 這個後台綁不綁獨特內部 workflow？（越綁特定流程、買的客製天花板越快撞到 → 越偏自建）
3. app 可攜性如何？（non-exportable 的設定 = 停付即失去所有工具 = lock-in）
4. RBAC / audit log / SSO 是基本盤就有、還是鎖在 Enterprise tier 才給？
5. 開發速度的機會成本：自建後台「數月」vs 低代碼「分鐘 / 天」期間、operator 被卡住的代價多大？
6. vendor 故障時後台不可用、營運中斷面多大？（internal tool 掛掉 = operator 停擺、不像對外服務有現成 SLA 賠償、這塊的可用性風險常被當「內部的沒關係」低估）

問完一塊能力的問題集後、把結論落回兩個地方再進下一塊：domain map 那一列的「自建 / 外包」欄填上判斷、決策記錄補一條這塊的整合接縫（webhook / API 由誰維護、資料一致性怎麼保）。每塊都過完、回到上面的收斂判準逐條確認 —— 這條路由讓問題集收斂回 domain map、不停在「問完」。

## Event 的 SRP 判準

一個 event 一個事實、過去式命名：

1. **一個事實**：`OrderPlacedAndPaid` 是兩個事實 — 下單成立與付款完成可以相隔數分鐘、失敗路徑完全不同、訂閱者也不同、拆成兩個 event。
2. **過去式、不可否認**：event 是「已發生」、發出後就是歷史。`OrderPlaced`、`PaymentFailed`、`MemberRemoved`。
3. **命令偽裝偵測**：名字是祈使句的「event」（`SendWelcomeEmail`）實為 command — 它指定了誰該做什麼、發出者知道太多。改成事實（`UserRegistered`）、讓通知 domain 自己決定訂閱後要寄什麼。這個偵測是 OCP 的前哨：命令式 event 等於跨 domain 的隱性呼叫。
4. **失敗是一級事實**：`PaymentFailed` 跟 `PaymentSucceeded` 同等地位、有自己的訂閱者（通知使用者、暫停服務、進催收流程）。把失敗留在 exception 裡、失敗的後續處理就永遠掛不上骨架。

### 開發者工具 / 開源工具的 commodity check 簡化版

開發者工具的 commodity domain 清單和 SaaS 產品不同 — 不問 SSO、不問金流、不問表單。開發者工具常見的 commodity 能力和判讀問題：

| Commodity 能力 | 判讀問題                                                           |
| -------------- | ------------------------------------------------------------------ |
| 認證           | 要不要 HTTP auth？同區網信任還是公網暴露？API key 還是帳號系統？   |
| 通知           | 規則命中時怎麼通知？shell command（ntfy / curl）夠嗎？需要 email？ |
| 搜尋 / 查詢    | grep / jq 夠嗎？需要索引嗎？SQLite 的升級路徑是什麼？              |
| 儲存           | 檔案系統夠嗎？需要 DB 嗎？什麼規模觸發升級？                       |
| CI / 測試      | 開源專案預設要 CI。GitHub Actions 免費額度夠嗎？                   |

開發者工具的 commodity check 更常命中「自建」— 因為整個工具的設計哲學通常是零外部依賴、能用單一 binary 跑起來。「買掉」在這裡的意思更多是「用系統工具取代」（用 cron + curl 取代通知 SaaS、用 grep 取代搜尋服務）、而非接入第三方 SaaS。

## OCP：公開面與內部面

OCP 在領域層的意思：**明確區分別的 domain 需要知道的東西、跟不需要知道的東西** — 對擴充開放（別人可以訂閱你的事實、查你的公開介面）、對修改封閉（你的內部怎麼改、別人不需要重新編譯他的認知）。

每個 domain 維護兩張清單：

| 面                                   | 內容                                                | 變更紀律                                                               |
| ------------------------------------ | --------------------------------------------------- | ---------------------------------------------------------------------- |
| **公開面**（別的 domain 需要知道）   | event 名稱與 payload schema、公開查詢介面（讀取用） | 這是 contract — 變更要盤點所有訂閱者、「只加欄位、不改不刪」是預設姿態 |
| **內部面**（別的 domain 不需要知道） | 表結構、狀態機、內部計算規則、私有欄位              | 可自由改、改了不需要通知任何人                                         |

判讀問題：**「別的 domain 為了完成自己的職責、最少需要知道什麼？」** 公開面刻意最小化 — event payload 帶「事實本身 + 訂閱者需要的最小資料」、把整個內部物件序列化進 payload 是最常見的洩漏：內部物件進了 contract、之後每次改表都在 break 訂閱者、OCP 邊界名存實亡。

跨 domain 協作的兩條路、優先序固定：

1. **Event 優先**：A 宣告事實、B 自己決定反應。A 不知道 B 存在、B 消失或新增都不影響 A — 擴充不需要修改。
2. **公開查詢介面**：需要立即答案的同步讀取（下單時查商品現價）走 domain 的公開查詢、回傳公開 schema、而不是直接 join 對方的表。

直接呼叫對方內部（import 對方的 model、join 對方的表）是邊界違規 — MVP 的單體 repo 裡這個違規零成本、所以要靠紀律與目錄結構（per-domain module）擋、這也是 scaffold 建議要落實的一項。

## 反向問

1. 「**Order domain 的表加一個欄位、要通知誰？**」名單越長、內部面洩漏越嚴重；理想答案是「不用通知任何人」。
2. 「**這個 event 砍掉、哪些功能會壞？**」沒有任何訂閱者的 event 是憑空想像的需求、先移出 catalog。
3. 「**未來最可能變的業務規則是什麼？它被關在一個 domain 裡了嗎？**」定價、佣金、配額這類高頻變動規則散落多個 domain、就是切分要重看的證據。
4. 「**這個 command 失敗一半會怎樣？**」一個 command 的處理跨越兩個 domain 的寫入、表示 transaction 邊界跨 domain — 重切、或改成 event 驅動的最終一致、當場依下列條件判讀：
   - 資源是庫存 / 容量 / 餘額類 — 超量有真實業務代價（課堂超賣、餘額透支）→ 強一致傾向：同 transaction 完成、或把資源持有權併進發起 command 的 domain。
   - 資源是通知 / 統計 / 投影類 → 最終一致 + 補償。
   - 重切 vs 維持兩 domain 的分界：同一條業務規則改動是否同時動兩個 domain — 是、變更理由其實同一個、合併；否、維持兩 domain 走 event。

## 產出格式

**Domain map**、每個 domain 一列（「自建 / 外包」欄由 commodity domain check 填）：

| Domain | 責任（一句話、一個變更理由） | 自建 / 外包 | 處理的 commands | 公開面（events + 查詢介面） | 內部面要點 |
| ------ | ---------------------------- | ----------- | --------------- | --------------------------- | ---------- |

**Event catalog**、每個 event 一列：

| Event | 來源 domain | Payload contract（最小欄位） | 訂閱者 | 失敗語意（可丟 / 不可丟） |
| ----- | ----------- | ---------------------------- | ------ | ------------------------- |

「失敗語意」欄的判讀來源是操作盤點的失敗情境與風險欄：操作牽涉金流、合約或通知義務 → 它的 event 不可丟；event 只驅動可重算的衍生結果（統計、推薦、快取更新）→ 可丟。這一欄直接餵維度展開的 async-queue 維度（不可丟 event 觸發 durable 機制與 outbox）；domain map 的邊界餵 state-storage（transaction 邊界沿 domain 配置）與 security（授權沿 domain 邊界檢查）。

收斂判準（全部通過才進核心問題）：

1. 操作清單上的每個操作都有歸屬 domain、且歸屬唯一。
2. 每個 domain 都過了 commodity domain check：自建的標差異化理由、買掉的標「外包 + 整合邊界」、沒有「現成 SaaS 已做完卻仍排進自建」的殘留。
3. 每個跨 domain 協作都有對應 event 或公開查詢介面、沒有「直接讀對方的表」的殘留。
4. 每個 domain 的公開面寫得出來且刻意最小；寫不出來表示這個 domain 的責任還說不清楚。
5. 每個 event 通過 SRP 檢查：一個事實、過去式、有訂閱者、失敗語意已標。買掉的 domain 不在自己的 catalog 模內部 event、只在邊界處理 vendor 的 webhook。
