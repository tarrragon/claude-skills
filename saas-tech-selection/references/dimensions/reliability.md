# Dimension：Reliability（驗證流程與依賴可靠性）

本維度處理「壞的版本怎麼在使用者看到之前被攔下、依賴失敗時系統怎麼不跟著一起死」。必展開、day one 形態跟觀測一樣是底線集合：CI gate、回滾演練、第三方依賴的失敗語意。完整的可靠性工程（load test、chaos、contract test）是規模觸發項。

---

## 訪談問題

| 問題                                                   | 為什麼問                                                           |
| ------------------------------------------------------ | ------------------------------------------------------------------ |
| 依賴哪些第三方服務（付款、email、簡訊、外部 API）？    | 每個第三方都是一個失敗模式 + 一個費用模型 + 一個 sandbox 需求      |
| 第三方掛掉時、對應功能要排隊重試、降級、還是直接報錯？ | 失敗語意是設計決策、放著不設計就會變成「整個 request 跟著掛」      |
| 哪些操作絕對不能執行兩次（扣款、開通、寄合約）？       | 圈出 idempotency 的硬需求範圍 — 重試機制與重複執行防護必須成對出現 |
| 測試打算寫到什麼程度？團隊有寫測試的習慣嗎？           | 決定 CI gate 的起始門檻 — 門檻設在團隊會遵守的高度、設太高會被繞過 |

**反向問**：「付款服務回應 timeout、錢到底扣了沒有？你的系統現在記成什麼？」— 第三方 timeout 的曖昧狀態（可能成功可能失敗）需要對帳或查詢機制收斂、這是金流相關 SaaS 最高頻的事故源。

## Day one 最小集

| 機制           | 最小實作                                                                                                                       |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| CI gate        | 測試 + lint 通過才能 merge；lockfile 進 repo、CI 用鎖定版本安裝                                                                |
| 測試起始集     | **三層策略**：unit（mock）/ protocol integration（真實服務）/ screen state（UI），見 `principles/three-layer-test-strategy.md` |
| 第三方呼叫紀律 | 每個外部呼叫都有 timeout、可重試的配 backoff、不可重試的記錄待對帳                                                             |
| Idempotency    | 不能執行兩次的操作配 idempotency key（DB unique 約束是最簡單的實作）                                                           |
| 回滾演練       | 部署回滾在 staging 或低風險時段實際執行過一次                                                                                  |
| 環境 parity    | 本地 / CI / production 用同一套 container 定義或依賴鎖定、降低「只有線上會壞」                                                 |

測試起始集的三層策略：unit test 覆蓋邏輯分支（mock 外部依賴）、protocol integration test 覆蓋真實服務互動（至少涵蓋 event catalog 中的協議路徑）、screen state test 覆蓋 UI 狀態。Unit test 的 mock 遮蔽了協議層行為 — 加更多 mock test 只增加同層覆蓋率、不會發現協議不相容或認證缺失。名稱含「integration」但依賴全用 fake 的 test 要辨識並標明真實驗證邊界。

第三方依賴每接一個、決策記錄記四件事：sandbox 環境怎麼拿、失敗語意（重試 / 降級 / 報錯）、費用模型與用量上限 alert、webhook 的驗簽方式。第三方的 webhook 是入口維度之外的另一個對外面、驗簽缺席等於開了一個免認證寫入口。

---

## 防護底線（non-negotiable）

1. **Lockfile + CI gate**：依賴鎖定讓環境可重現、CI 讓「壞了會被攔」有最低保證。
2. **外部呼叫全部有 timeout**：沒有 timeout 的外部呼叫、會把第三方的慢轉成自己的連線耗盡。
3. **重試與 idempotency 成對**：先有重複執行防護、才允許自動重試；順序反過來就是重複扣款事故。
4. **Webhook 驗簽**：接收第三方 webhook 必驗簽名、必做重放防護（timestamp 容忍窗）。

## 延後與 tripwire

| 可延後項          | 建議重評條件                                           |
| ----------------- | ------------------------------------------------------ |
| Load test         | 首次可預期高峰活動前、或容量規劃需要數字依據時         |
| Contract test     | 服務拆分後、跨服務 API 變更開始互相打破                |
| Chaos / 故障演練  | 依賴數量與失敗路徑多到人工推理蓋不住                   |
| Feature flag 系統 | 需要漸進釋出或大功能長期分支造成合併痛苦               |
| Circuit breaker   | 某個依賴的失敗開始造成連鎖失敗（重試風暴、執行緒耗盡） |

---

## 決策記錄要記什麼

CI gate 構成與門檻、測試起始集範圍、第三方依賴清單（每個：sandbox / 失敗語意 / 費用 alert / 驗簽）、idempotency 適用operations 清單、防護底線段各條的狀態、延後項觸發條件。
