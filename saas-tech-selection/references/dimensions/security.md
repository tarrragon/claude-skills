# Dimension：Security（身份、隔離、secret、資料保護、追溯）

本維度處理「誰能對哪些資料做什麼、怎麼證明」。安全在初始化階段的特殊性：它是事後補建成本最高的維度 — 權限模型重構要動每個 endpoint、audit 紀錄無法回溯補登、外洩發生後才建的控制只剩公關價值。所以本維度必展開、且底線密度最高。

訪談姿態：安全問題用具體情境問（「客服能看到客戶的付款資料嗎」）、用名詞問（「要不要 RBAC」）使用者答不出來。

---

## 訪談問題

| 問題                                              | 為什麼問                                                                      |
| ------------------------------------------------- | ----------------------------------------------------------------------------- |
| 使用者怎麼登入 — email 密碼、社群登入、企業 SSO？ | B2B 的 SSO 需求決定要不要 day one 就接 OIDC / SAML、事後接是賣單門檻          |
| 系統內有哪幾種角色？各自能看 / 改什麼？           | 圈出授權模型的最小集合；角色 ≤ 3 個時簡單 role check 即可、過度設計同樣有成本 |
| 內部人員（客服、工程師）怎麼存取客戶資料？        | 內部存取是最常被跳過的面 — 後台權限、production DB 直連、log 裡的個資         |
| 哪些資料是個資 / 敏感資料？服務哪些地區的使用者？ | PII 盤點驅動 log 遮罩與保留政策；地區決定合規框架（GDPR 類）的適用            |
| 使用者要求刪除資料時要做到什麼程度？              | 刪除義務牽動備份策略與第三方資料殘留、day one 想過跟沒想過差很多              |

**反向問**：「這批資料全部外洩、最壞情境是什麼 — 道歉信、賠償、還是公司結束？」— 答案決定本維度的投資等級；B2B 多租戶答案通常是最後者、底線等級隨之上調。

## 子領域與選型

### 身份認證（authentication）

**Day one 預設**：用框架成熟方案或託管身份服務（Auth0 / Cognito / Firebase Auth 類）、把密碼雜湊、session 管理、重設流程、MFA 外包。自建認證的合理場景很少 — 確認使用者要自建時、把「密碼外洩責任自負」攤開講。

**B2B 注意**：企業客戶遲早問 SSO 與 SCIM；選託管服務時確認這條路存在、避免換服務重做。

**單人自架工具**：服務只有所有者使用時、認證不走 web-auth（帳號 / session / SSO）、改用**裝置綁定 + 共享密鑰**兩層 — 裝置原生生物辨識（Face ID / BiometricPrompt）認「人」、app 與服務端共享密鑰認「連線」。密鑰用裝置安全儲存（Keychain / Keystore）、不硬寫進 app（反編譯可挖）;配對走 air-gap 通道（QR 掃描）避免經網路 / 剪貼簿傳輸。失效模型跟多人 session 不同：裝置失竊等於認證邊界整個失效、無中央會話可遠端 kill、唯一收斂是服務端輪替密鑰版本強迫重新配對。

### 授權（authorization）

**Day one 預設**：角色檢查（role check）+ 物件層級擁有權檢查。兩層缺一不可 — 角色檢查擋「客服不能進管理功能」、物件檢查擋「使用者 A 改 URL 看使用者 B 的訂單」。後者（BOLA）是 SaaS 最常見的實際外洩路徑、且測試最容易漏。

有 domain map 的場合、授權配置沿 domain 邊界落點：每個 domain 對自己的 command 做擁有權檢查、跨 domain 的存取只能走公開查詢介面（介面內建租戶 / 擁有權過濾）— 授權邏輯散落在 handler 各自為政、就是邊界沒立住的訊號。

### 租戶隔離

多租戶 SaaS 的隔離是安全與資料兩維度的交集：查詢層強制 scope（詳見 state-storage 維度）、加上 API 層的租戶上下文驗證（token 內的租戶聲明對 URL / payload 內的租戶 id）。兩層各自獨立失效、疊加後漏網率才可接受。

### Secret 管理

Day one：`.env` + `.gitignore` + `.env.example`、CI / 部署平台的 secret 注入、commit 前掃描工具。已知的輪換時機（離職、外洩疑慮）寫進 README。Secret manager 服務在 secret 數量或存取主體變多時引入。

### 資料保護

PII 欄位盤點（哪幾張表哪幾欄）、log serializer 遮罩這些欄位、傳輸全程 TLS。加密儲存（at-rest）託管平台多半內建、確認開啟即可。保留與刪除：每類資料寫一行保留期限、刪除請求的處理流程（含備份內資料的處理立場）寫進決策記錄。

### Audit 與追溯

Day one 最小集：部署紀錄、管理操作 log（誰改了誰的資料）、production DB 直連的紀錄與審批立場。完整 audit log 系統可延後、但「完全沒有紀錄」狀態每多活一天、就多一天無法回答「資料怎麼變這樣」。

## 防護底線（non-negotiable）

1. 對外 endpoint 預設要求認證、公開路由白名單放行（預設拒絕、顯式開放）。
2. 物件層級授權檢查進框架層（middleware / base query）、不依賴每個 handler 各自記得。
3. 輸入驗證 + 欄位白名單綁定（mass assignment 防護）。
4. Secret 不進 repo、TLS 全程 + 自動續期。
5. PII 欄位盤點完成、log 遮罩就位。
6. 管理操作留下紀錄。
7. Operator 帳號（雲 console、DB、部署平台、網域註冊商）全開 MFA — 一組管理憑證外洩等於全盤失守、成本接近零。

## 延後與 tripwire

| 可延後項             | 建議重評條件                                  |
| -------------------- | --------------------------------------------- |
| WAF / 進階入口防護   | 公開面擴大、或出現自動化攻擊流量              |
| 完整 audit log 系統  | 合規要求、或第一個企業客戶的安全問卷          |
| Secret manager 服務  | secret 數量 > 10、或存取主體（服務 / 人）> 5  |
| 滲透測試             | 上線後首次、之後隨重大架構變更                |
| 單人認證升級帳號系統 | 從單人 / 自用變成多使用者、或要分享存取給他人 |

## 決策記錄要記什麼

認證方案（託管 / 框架 / 自建 + 理由）、角色與授權模型、租戶隔離的查詢層與 API 層機制、PII 盤點結果與遮罩位置、刪除請求處理立場、防護底線段各條的狀態、延後項與重評條件。
