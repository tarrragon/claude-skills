# Dimension：Async Queue（非同步交接與事件傳遞）

本維度處理「request 之外要可靠完成的工作」— 它是 event catalog 的執行層。訪談的主要輸入是 domain / event 切分階段產出的 event catalog：每個 event 已經標好來源 domain、訂閱者與失敗語意（可丟 / 不可丟）、本維度負責回答「這些 event 用什麼機制送、怎麼保證語意」。catalog 中存在不可丟 event 時、本維度直接升級為必展開；尚未做 domain / event 切分的場合、退回用工作清單（寄信、webhook、報表、對帳）盤點。

選型的第一個區分沿用 catalog 的失敗語意欄：**可丟（重算就好）的 event 用最簡單的背景 job；不可丟（錢 / 合約 / 通知義務）的 event 才值得 durable queue 與 outbox 的複雜度**。

---

## 訪談問題

| 問題                                                                                        | 為什麼問                                                                     |
| ------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| event catalog 之外、還有哪些 request 外的工作（排程、對帳、清理）？                         | catalog 收的是領域事實、純技術性的背景工作（憑證續期、過期清理）要在這裡補盤 |
| 每個不可丟 event 跟來源 domain 的資料寫入要不要同生共死？（OrderPlaced 必須跟訂單寫入一致） | 要 → outbox 模式進場；DB 寫入跟發訊息分開做、就有「寫入成功但事件丟了」窗口  |
| 同一個工作被執行兩次會怎樣？                                                                | at-least-once 是預設世界觀 — 重複投遞必然發生、消費端要 idempotent           |
| 工作量級與延遲容忍？（每分鐘幾個、可以等多久）                                              | 量級小 + 容忍高的場景、DB-backed queue 就夠、broker 是過度配備               |
| 即時推送（聊天、通知）是硬需求嗎？                                                          | 推送走 pub/sub + 長連線、跟可靠工作佇列是不同機制、混用會兩邊都做不好        |

**反向問**：「寄信服務連續失敗一小時、這一小時的信會怎樣 — 補發、丟掉、還是塞爆重試把服務拖垮？」— 暴露重試策略、退避、與死信的設計缺口。

## 候選類型差異

| 類型                                                   | 適合                                       | 代價                                               |
| ------------------------------------------------------ | ------------------------------------------ | -------------------------------------------------- |
| 框架背景 job（DB-backed：Sidekiq / Celery / River 類） | day one 的寄信、報表、清理工作             | 吞吐上限低於專用 broker、但 MVP 階段遠遠夠用       |
| Durable queue（RabbitMQ / SQS 類）                     | 不可丟工作、量級上來、多消費者             | 多一個有狀態元件；SQS 類託管可消掉維運面           |
| Stream（Kafka / Redis Streams 類）                     | 事件要重放、多下游各自消費、事件是資料資產 | 認知負擔與維運成本高、MVP 階段幾乎不該出現         |
| Pub/sub + 長連線                                       | 即時推送（線上通知、聊天）                 | fire-and-forget — 不在線就丟、可靠通知要另外補存底 |

**SaaS day one 預設**：框架原生背景 job（用既有 DB 當 backing store、零新元件）+ 對「不可丟」工作套 outbox 思路（工作記錄與業務資料同 transaction 寫入、worker 輪詢處理）。專用 broker 的進入條件寫成 tripwire：job 表的輪詢開始壓 DB、或吞吐 / 延遲要求超出 DB-backed 上限。

---

## 防護底線（non-negotiable）

1. **消費端 idempotent**：重複投遞是常態、每個 worker 要嘛天然冪等、要嘛配 idempotency key 檢查。
2. **重試有上限 + 退避 + 死信去處**：無上限重試會把暫時故障放大成重試風暴；重試耗盡的工作進死信（最簡：標記失敗的表 + alert）、而不是無聲消失。
3. **不可丟工作與業務寫入同 transaction**：outbox 模式或同等機制、消除「資料成立但事件遺失」窗口。
4. **佇列深度進觀測**：積壓（backlog）是下游故障的最早訊號、深度與最老工作年齡兩個數字配 alert。

---

## 延後與 tripwire

| 可延後項     | 建議重評條件                                     |
| ------------ | ------------------------------------------------ |
| 專用 broker  | job 輪詢壓 DB、或單表 job 吞吐撐不住             |
| Stream 平台  | 出現「多下游重放同一事件源」的真實需求           |
| 排程系統升級 | cron 單點開始造成漏跑 / 重複跑、需要分散式排程鎖 |

工作清單（每個：可丟 / 不可丟、延遲容忍、失敗語意）、機制選型與理由、outbox 適用範圍、重試 / 死信策略、防護底線段各條的狀態、broker 升級觸發條件。
