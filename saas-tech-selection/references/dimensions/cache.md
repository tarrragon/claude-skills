# Dimension：Cache（快取與暫存副本）

本維度處理「重複讀取與昂貴計算的結果、值不值得保留一份副本」。觸發展開：操作清單或核心問題出現高頻重複讀、昂貴計算共用、session / presence 需求時才進入。MVP 階段最常見的正確答案是「先不要」— 快取是用一致性換速度的交易、在還沒有讀壓力時引入、只買到一致性問題沒買到速度。

---

## 訪談問題

| 問題                                                        | 為什麼問                                                             |
| ----------------------------------------------------------- | -------------------------------------------------------------------- |
| 哪些資料被重複讀最多？讀到舊值的容忍度是多少（秒 / 分鐘）？ | 新鮮度容忍是快取設計的第一參數 — 容忍度為零的資料快取成本極高        |
| 是讀多、還是算貴？（同一查詢重複跑 vs 一次計算很貴）        | 讀多 → cache aside；算貴 → 結果快取 + 過期重算策略                   |
| Session 放哪裡？                                            | session 進外部 store 是水平擴展前提、跟快取常共用同一個 Redis        |
| 有沒有「誰在線上」「即時計數」類需求？                      | presence / counter 是 Redis 資料結構的場景、跟讀加速是不同的需求類型 |

**反向問**：「快取整層掛掉、流量全部直接打到 DB、撐得住嗎？」— 快取引入的同時要想好它消失的那天（cache 故障、大量同時過期、部署後冷啟動）、回源風暴的防護跟快取本體同時設計。

---

## 候選類型差異

| 類型                   | 適合                                    | 代價                                     |
| ---------------------- | --------------------------------------- | ---------------------------------------- |
| In-process（記憶體內） | 單實例、小量熱資料、容忍實例間不一致    | 多實例下各自為政、失效廣播要另外做       |
| Redis（外部快取）      | 多實例共享、session、presence、分散式鎖 | 多一個要維運的有狀態服務、序列化成本     |
| HTTP / CDN 層快取      | 整頁或 API response 可快取、匿名流量大  | 個人化內容要切 vary 維度、失效控制粒度粗 |

**引入順序建議**：HTTP 層（最便宜、零新元件）→ in-process（一行 LRU）→ Redis（確認需要跨實例共享才上）。使用者點名 Redis 時、確認需求是「跨實例共享」還是「覺得該有快取」— 後者先回到讀壓力證據。

**外部託管判讀**：自管 Redis 最容易被低估的是有狀態服務的維運 —— 備援、failover、記憶體壓力都得自己扛（深度選項見 `principles/capability-outsourcing-depth.md`）。同一塊快取有三種深度：自管 Redis、managed cache（ElastiCache / Upstash serverless 類、把維運外包）、或折進跨能力 bundle 自帶的快取 / KV。小團隊預設 managed —— 快取掛了回源風暴的處理已經夠燒、再扛一個有狀態服務的維運多半不划算。若 state-storage 已選了某個 bundle、先確認它的快取能力夠不夠、避免多接一個獨立 Redis 又多一道接縫。

## 防護底線（non-negotiable）

1. **快取永遠可丟**：快取內容必須可從正式狀態重建、任何業務資料以快取為唯一儲存點就是資料遺失事故的排程。
2. **每個 key 都有 TTL**：沒有 TTL 的快取是記憶體洩漏 + 永不過期的舊值；TTL 是失效策略的保底、不是失效策略本身。
3. **失效策略先於引入**：寫入時更新還是刪除、批次資料變更時怎麼清 — 寫不出失效策略的快取先不要上。
4. **回源保護**：熱 key 過期的同時湧入（stampede）用單飛（singleflight）或短期鎖防護；快取層故障的回源要有限流。

## 延後與 tripwire

| 情境                                             | 建議動作                                               |
| ------------------------------------------------ | ------------------------------------------------------ |
| 尚未引入快取                                     | DB 讀延遲與讀寫比進觀測、讀壓力證據出現再回到本維度    |
| 已引入、命中率低於起始門檻（50% 起、依定錨調整） | 快取的 key 設計或 TTL 跟存取模式不合、重看而不是加容量 |
| 快取與 DB 不一致頻發                             | 失效策略與寫入路徑對不上、收斂寫入口                   |

---

## 決策記錄要記什麼

引入與否的證據（讀壓力數字或「未引入 + 觸發條件」）、快取層級選擇、新鮮度容忍表（每類資料的 TTL 依據）、失效策略、回源保護機制、防護底線段各條的狀態。
