# Baseline Protections：不可沉默跳過的防護底線

本檔是跨維度的防護底線總清單與延後記錄協議。底線的定義：**缺了它、第一次事故的代價會遠超過 day one 建立它的成本**。底線跟選型不同 — 選型有多個合理答案、底線只有「已納入 / 已告知後延後 / 不適用」三種狀態、且三種狀態都要進決策文件。

訪談時底線逐項過、不靠使用者主動想到。使用者沒問到的底線正是這份清單存在的理由。

---

## 底線清單

### 一、Secret 與憑證

| 底線                  | 最小實作                                                   | 缺了會發生什麼                                         |
| --------------------- | ---------------------------------------------------------- | ------------------------------------------------------ |
| Secret 不進 repo      | `.env` + `.gitignore` + `.env.example` 模板；commit 前掃描 | API key 進了 git 歷史、撤銷要 rotate 全部憑證 + 清歷史 |
| Secret 有單一管理位置 | 環境變數注入；規模化後進 secret manager                    | secret 散在 CI 設定、部署腳本、同事筆記、rotate 時漏改 |
| TLS 全程 + 自動續期   | 託管平台內建、或 ACME 自動簽發；憑證到期 alert             | 憑證過期是「可預防卻最常見」的全站停機原因之一         |
| Operator 帳號 MFA     | 雲帳號 root / DB console / 部署平台 / 網域註冊商全開 MFA   | 一組管理憑證外洩 = 全盤失守、防護成本接近零            |

### 二、入口與輸入

| 底線                            | 最小實作                                                       | 缺了會發生什麼                                               |
| ------------------------------- | -------------------------------------------------------------- | ------------------------------------------------------------ |
| 對外 endpoint 必有認證邊界      | 框架層 authentication middleware、預設拒絕、白名單放行公開路由 | 忘記掛 auth 的新 endpoint 直接裸奔、事後盤點困難             |
| 物件層級授權檢查                | 查詢資源時驗證「這個 id 屬於這個使用者 / 租戶」                | 改 URL 上的 id 就能看到別人的資料（BOLA）、SaaS 最常見外洩源 |
| 輸入驗證 + mass assignment 防護 | schema validation、白名單欄位綁定                              | 使用者塞 `is_admin=true` 進註冊 payload                      |
| Rate limit / 資源上限           | 入口層基本限流、上傳大小上限、分頁上限                         | 單一 client 打掛全站、或一個 `?limit=999999` 拖垮 DB         |

### 三、資料

| 底線                        | 最小實作                                                                                                  | 缺了會發生什麼                                       |
| --------------------------- | --------------------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| Schema migration 版本化     | migration 工具管 schema、禁止手改 production schema                                                       | 環境間 schema 漂移、部署不可重現、回滾無依據         |
| 自動備份 + 至少一次還原驗證 | 託管 DB 開自動備份、備份存取權與 production 隔離（帳號淪陷 / 勒索時不同鍋）；上線前實際還原一次到測試環境 | 「有備份」跟「能還原」是兩件事、差異在事故當天才發現 |
| 租戶 / 使用者隔離邊界       | 多租戶：每條 query 強制帶租戶條件（ORM scope / RLS）                                                      | 跨租戶資料外洩、B2B 場景等於商譽終結                 |
| PII 盤點 + log 遮罩         | 標記含個資欄位、log serializer 排除或遮罩這些欄位                                                         | 個資進了 log 平台、保留期限與存取控制全部失守        |

### 四、部署與變更

| 底線               | 最小實作                                                      | 缺了會發生什麼                                               |
| ------------------ | ------------------------------------------------------------- | ------------------------------------------------------------ |
| 部署可回滾         | 保留上一版 artifact / image、回滾指令寫進 README 並演練過一次 | 壞版本上線後只能往前修、事故時間從分鐘變小時                 |
| Health check       | liveness / readiness endpoint、部署流程驗證通過才切流量       | 壞實例持續收流量、部署成功但服務已停止回應                   |
| 依賴鎖定 + CI gate | lockfile 進 repo、CI 跑測試 + lint 通過才能 merge             | 「我這邊可以跑」、production 裝到不同版本依賴                |
| 依賴漏洞掃描       | dependabot / audit 類自動掃描開啟、高危 CVE 出 alert          | lockfile 凍結的是版本、不是安全 — 已知漏洞依賴是頭部入侵向量 |

### 五、觀測與追溯

| 底線                      | 最小實作                                               | 缺了會發生什麼                                     |
| ------------------------- | ------------------------------------------------------ | -------------------------------------------------- |
| Structured log + 錯誤分類 | JSON log、區分 4xx / 5xx / 依賴失敗、帶 request id     | 事故時只能 grep 純文字、找不到因果鏈               |
| 掛了有人知道              | uptime 監控 + alert 送到會被看到的地方（手機通知層級） | 使用者比團隊先發現停機                             |
| Production 操作可追溯     | 部署紀錄、DB 直連紀錄（最小：誰、何時、做了什麼）      | 「資料怎麼變這樣」永遠查不出來、內部誤操作無法歸因 |

### 六、金流分帳（涉及分帳 / 抽成 / 平台代收時適用）

| 底線         | 最小實作                                                                       | 缺了會發生什麼                                                     |
| ------------ | ------------------------------------------------------------------------------ | ------------------------------------------------------------------ |
| 分帳對帳機制 | 平台流水對第三方金流流水的定期核對（排程比對或固定週期人工核對、差異出 alert） | 漏帳 / 重複入帳 / 抽成算錯在月結或客訴時才發現、追溯要逐筆人工比對 |

本項適用下方延後記錄協議、重評條件例：第一筆真實分帳發生前。

## 延後記錄協議

使用者對任一底線說「先跳過」「之後再說」時、依下列協議處理：

1. **告知代價**：一句話講清楚缺口的失敗情境（用上表「缺了會發生什麼」欄）、確認使用者理解的是代價、不只是名詞。
2. **轉寫成延後條目**：決策文件的防護底線總表該項標「延後」、附三個欄位 — 告知日期、使用者決定延後的理由、重評條件。
3. **重評條件要具體**：「之後補」不算條件；「上線前」「第一個付費客戶前」「資料筆數破十萬前」才算。skill 推薦一個條件、使用者可改。
4. **不適用要給理由**：純內部工具可以對「rate limit」標不適用、但理由（「僅 VPN 內網可達」）要寫進表、讓未來讀者知道前提何時失效。

延後協議的目的：讓六個月後接手的人能回答「當初為什麼沒做」、把口頭妥協變成可追溯的決策。

## 與維度選型的關係

每個 dimension reference 內也有該維度自己的防護底線段、內容與本檔部分重疊 — 這是刻意設計：維度訪談時就地確認該維度的底線、本檔在決策收斂階段當總表核對一次。兩次經過同一條底線、漏網率比單點檢查低。交付形態 gate 判託管的縮減流程跳過維度展開、底線只經過總表這一次 — 所以託管縮減記錄的防護底線總表（適用項）不可省略。
