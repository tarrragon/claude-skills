# Scale Stage Triggers：規模成長撞牆訊號與 tripwire 寫法

本檔回答兩個問題：系統規模成長時會依什麼順序撞牆、以及 repo 初始化階段怎麼把這些牆轉成決策文件的 tripwire 總表。核心立場是 **初始化時不為未來的牆預建能力、但要為每面牆預建偵測訊號與重評承諾** — 預建能力是過度設計、預建訊號是低成本保險。

---

## 典型撞牆階段

階段編號描述常見順序、不是嚴格時序。本檔的撞牆階段屬於自建世界；交付形態 gate 判託管的流程、tripwire 是另一套（平台內 workaround 累積、費用曲線交叉、客製被擋、配額影響業務、合規超出平台控制面）、來源是定錨段的交付形態 gate、不套用本檔階段。團隊可能跳階段、反覆、或永遠停在前兩階段（多數內部工具與小型 B2B 一輩子不需要階段 E、F）。

### 階段 A：單機資源吃滿

**撞牆訊號**：CPU 或記憶體持續高檔、回應時間隨流量線性惡化、重啟暫時緩解。

**該做的事**：先確認 stateless 前提（session、上傳檔案、排程鎖有沒有黏在本機）、再評估垂直擴（換大機器、簡單但有天花板）vs 水平擴（加機器、需要 stateless + 共享狀態外移）。

**初始化階段的對策**：把 session 與檔案從 day one 就放外部（DB / object storage）、讓水平擴的路徑保持暢通。這是少數值得「為未來預付」的設計、因為 day one 做幾乎零成本、事後改要動到認證與上傳流程。

### 階段 B：資料庫變慢、加機器無效

**撞牆訊號**：DB CPU 高但流量沒漲多少、特定頁面越用越慢、慢查詢集中在少數 query。

**該做的事**：先清應用層查詢反模式（N+1、缺索引、全表掃描、select 全欄位、未分頁列表）收回單機容量、再考慮升級規格。反模式清掉之前升規格只是把問題推遲且變貴。

**初始化階段的對策**：開啟慢查詢紀錄（slow query log）、ORM 加 query 數量的開發期警告。偵測在 day one 是零成本設定、修復可以等訊號出現。

### 階段 C：讀取壓爆 primary

**撞牆訊號**：寫入正常但讀取延遲飆高、讀寫比例嚴重失衡（常見 10:1 以上）、快取命中率低或沒有快取。

**該做的事**：先加快取層（cache aside）擋掉重複讀、再評估 read replica 與應用層讀寫路由。replica 引入 replication lag、「自己寫的資料馬上讀」的場景要留在 primary。

### 階段 D：靜態資源打爆 origin

**撞牆訊號**：活動或外部引流時、頻寬與連線數先於 CPU 撞頂、流量大宗是圖片 / JS / 可快取頁面。

**該做的事**：靜態與半靜態內容放到 CDN / 邊緣層、origin 只承擔動態請求；同時設 origin protection（cache 失效時的回源限流）避免 CDN 失效瞬間打死後端。

### 階段 E：同步呼叫互相阻塞

**撞牆訊號**：付款慢拖垮下單、寄信服務故障讓整個 request 失敗、跨服務呼叫鏈越來越長且任一環掛掉全鏈失敗。

**該做的事**：把「request 完成後才需要發生的事」拆成事件、走 queue 非同步化、配 idempotency 與重試。失敗代價高的交接（金流、訂單事件）用 outbox 確保資料與事件一致。

### 階段 F：團隊發版互相阻擋

**撞牆訊號**：發版要跨團隊協調、一個模組的 bug 擋住全部上線、資料表被多個業務領域共寫。

**該做的事**：沿真實邊界拆服務 — 邊界依據是資料所有權、團隊分工、部署節奏或流量特性四選一、不是「微服務比較先進」。有些團隊永遠不需要這一步。

## 初始化階段的 tripwire 總表寫法

決策文件的 tripwire 總表把上述訊號轉成「偵測手段 + 重評承諾」。每列三欄：撞牆訊號（要可觀測、不是感覺）、偵測手段（day one 就要有的訊號源）、觸發後重評什麼。

範例：

| 撞牆訊號                         | 偵測手段（day one 建立）        | 觸發後重評                       |
| -------------------------------- | ------------------------------- | -------------------------------- |
| 單機 CPU 持續 > 70% 超過一週     | 主機 metrics + alert            | 垂直 vs 水平擴展、stateless 盤點 |
| 慢查詢數量週增、p95 查詢 > 100ms | slow query log                  | 查詢反模式清理、索引盤點         |
| 讀寫比 > 10:1 且讀延遲惡化       | DB metrics 讀寫分拆             | 快取層引入、read replica 評估    |
| 頻寬 / 連線數在活動期間撞頂      | LB / 入口 metrics               | CDN 與 origin protection         |
| 單一 request 內第三方呼叫 > 2 個 | trace 或 request log 的依賴計數 | 非同步化、queue 引入             |
| 發版需要跨團隊排隊               | 發版頻率與阻塞紀錄（人工即可）  | 服務邊界重切                     |

寫表時的兩個原則：

1. **訊號要有數字門檻**：「DB 變慢」無法觸發任何行動、「p95 > 100ms 持續一週」才能。門檻數字依定錨答案調整、寫一個可改的起始值比留白好。定錨階段給不出量級數字時、起始值照上表範例值填、該列明文標註「未定錨、上線後第一次回顧用實測值校正」— 照範例值填是規則、不留白、也不靠訪談者自行發明數字。
2. **偵測手段要回指觀測底線、方向單向**：tripwire 表裡每個偵測手段、都應該對應觀測維度已建立的訊號源；偵測手段不在 observability 的 day-one 最小集時、把對應 metric 加進最小集 — tripwire 決定 observability 底線、不反向拿觀測現況回推刪減 tripwire、避免兩份文件互相回推的迴圈。
