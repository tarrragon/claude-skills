# Decision Record Template：設計決策記錄與 scaffold 建議格式

本檔是決策收斂階段（訪談最終站）的產出規格。決策記錄的責任是讓三種讀者各取所需：現在的使用者確認「訪談結論跟我想的一樣」、未來的維護者回答「當初為什麼這樣選」、規模成長時的重評者知道「什麼訊號出現該回來改」。記錄的骨架順序跟訪談一致：操作與風險（BDD）→ domain map 與 event catalog（DDD）→ 技術維度決策 — 技術選型永遠能往上回溯到它服務的操作與領域。scaffold 建議是決策記錄的下游 — 先確認決策、再產生檔案。

---

## 決策記錄模板

```markdown
# 技術選型決策記錄 — <專案名>

產出日期：<date>
訪談依據：saas-tech-selection skill v<version>

## 0. 專案定錨

- 產品形態：<B2B / B2C / 內部工具>
- 租戶模型：<單租戶 / 多租戶（隔離層級）>
- 規模假設：<首年用戶量級、資料量級>
- 團隊能力：<人數、維運經驗>
- 上線時程：<時程與迭代節奏>
- 成本模型：<最貴的資源是什麼、月預算量級>
- 交付形態：<自建 / 託管平台 / BaaS / 半託管 CMS / 混合（流程落點見下表）+ gate 判讀依據>

混合形態時、本段附流程落點總表 — 每條業務流程一列、標託管或自建：

| 業務流程 | 落點（託管 / 自建） | 判讀依據 |
| -------- | ------------------- | -------- |

（交付形態 gate 結論是全託管形態時、本記錄換成託管縮減記錄：專案定錨段照寫、後續段落換成「平台選擇與理由 / 可遷出保險狀態表 / 升級自建 tripwire」、防護底線總表保留適用項 — 帳號安全、operator MFA、資料匯出管控、自有網域。）

（混合形態 — 部分流程託管 + 部分自建 — 時、自建流程照本模板完整段落走；每條託管流程在「技術維度決策」段掛一個獨立小節、小節內含四項：平台選擇與理由、可遷出保險狀態、升級自建 tripwire、重評條件。託管小節的升級自建 tripwire 同步列入「規模成長 tripwire 總表」段、與自建流程的規模撞牆訊號並列。）

## 1. 使用者操作與風險表

（操作盤點階段的產物。每個操作主體一個小節、表格每列七欄）

| 操作 | 角色 | 主情境 | 失敗情境 | 風險類型 | 前端引導 | 後端防護 |
| ---- | ---- | ------ | -------- | -------- | -------- | -------- |

範圍外操作（訪談中浮現、明確不做的）獨立列出 + 不做的原因 — 它們是下一版需求討論的起點。

## 2. Domain Map 與 Event Catalog

（domain / event 切分階段的產物）

### Domain Map

| Domain | 責任（一個變更理由） | 自建 / 外包 | 處理的 commands | 公開面（events + 查詢介面） | 內部面要點 |
| ------ | -------------------- | ----------- | --------------- | --------------------------- | ---------- |

（commodity domain check 命中、買掉整塊的 domain：「自建 / 外包」欄標「外包（vendor）」、公開面寫「與 vendor 的整合介面」、內部面寫「無（vendor 持有）」；這些 domain 不進技術維度的完整展開、只走 reliability 維度的第三方依賴訪談。買掉的每塊另在「規模成長 tripwire 總表」之外補一條整合接縫的維護與一致性承擔者。）

### Event Catalog

| Event | 來源 domain | Payload contract | 訂閱者 | 失敗語意（可丟 / 不可丟） |
| ----- | ----------- | ---------------- | ------ | ------------------------- |

### 介面契約（無 event 流時必填）

透明轉發、反向代理、或無領域事件流的架構、契約不藏在 event payload 裡（Event Catalog 為空或不適用）— 此時本段補記 client ⇄ server 共用的介面契約、否則下游 spec 的「資料模型 / 介面規格」章節會無源、得手動重述（這是「伺服器但非架站」場景的常見缺口）。

| 契約項 | 形態（payload schema / 子協議 / 資料模型） | 雙方（誰產生、誰消費） | 版本標記 |
| ------ | ------------------------------------------ | ---------------------- | -------- |

（有完整 event 流的架構、契約已在 Event Catalog 的 Payload contract 欄表達、本段標「N/A — 契約在 event payload」即可。）

## 3. 技術維度決策

（每個展開過的維度一段、未展開維度收在本段最後；每段的需求判讀要回指使用者操作與風險表、以及 event catalog）

### <維度名>

- **需求判讀**：<操作盤點 / domain / event 切分 / 核心問題得到的具體需求、引用使用者的回答>
- **選型**：<能力類型 → 具體服務>（例：relational DB → managed PostgreSQL）
- **外包深度**：<self-host / managed 基礎設施 / feature SaaS / 折進 bundle X>（多維度共用同一 bundle 時、標明這塊屬哪個 bundle、整包遷出代價記在 tripwire）
- **理由**：<為什麼這個選型、回指定錨；列被淘汰的次選項與淘汰原因>
- **防護要求（已納入）**：<該維度 non-negotiable 清單中本次直接做的項目>
- **延後項**：<已告知、使用者選擇延後的項目 + 重評條件>
- **Tripwire**：<什麼訊號出現時這個選型要重評>

### 未展開維度

| 維度   | 未展開原因             | 觸發條件（出現時回來展開）   |
| ------ | ---------------------- | ---------------------------- |
| <維度> | <訪談各階段無對應訊號> | <具體訊號>                   |

## 4. 防護底線總表

| 底線                       | 狀態                  | 備註（延後理由 / 重評條件 / 不適用前提） |
| -------------------------- | --------------------- | ---------------------------------------- |
| Secret 不進 repo           | 已納入                |                                          |
| 自動備份 + 還原驗證        | 延後（2026-06-11 告知） | 第一個付費客戶前完成還原演練            |
| ...                        | ...                   | ...                                      |

## 5. 規模成長 tripwire 總表

| 撞牆訊號（含數字門檻） | 偵測手段（day one 建立） | 觸發後重評       |
| ---------------------- | ------------------------ | ---------------- |
| ...                    | ...                      | ...              |

（混合交付形態時、兩套 tripwire 並列本表：自建流程掛規模撞牆訊號、託管流程掛升級自建 tripwire — 後者來源是「技術維度決策」段各託管流程小節、每列標注來源流程與形態、與小節內容保持互指一致。）

## 6. Scaffold 清單

（格式見「Scaffold 建議格式」節、決策確認後填入）
```

### 填寫規則

1. **理由欄必須回指定錨或領域骨架**：「因為 PostgreSQL 比較好」不合格；「因為團隊兩人無維運經驗、託管服務把備份與升級外包、月成本在預算內」合格；「因為 Billing domain 的 PaymentFailed 是不可丟 event、需要 outbox + durable 投遞」合格。
2. **次選項要留名字**：未來重評時、被淘汰的選項是第一批回看對象；淘汰原因消失（例：團隊長出維運能力）就是重評入口。
3. **每項選型三欄齊備**：理由、防護狀態、tripwire 缺一不可 — 缺理由的決策無法重評、缺防護狀態的決策藏著沉默缺口、缺 tripwire 的決策會超期服役。
4. **使用者的原話值得保留**：需求判讀欄引用使用者在訪談中的具體說法（「重複扣款絕對不行」）、比轉述（「一致性要求高」）更能讓未來讀者還原當時的權重。
5. **§1 操作風險表、§2 Domain Map 是強制段、不可空白交付**：只填 §3 技術維度、跳過 §1 / §2 的決策記錄、是「訪談沒走完 Stage 1 / Stage 2」的半成品 — §3 的需求判讀無從回指（填寫規則 1 失效）、下游 doc 系統長不出 usecase（§1 為空）與 domain spec（§2 為空）、未來重評也讀不到當初的操作與領域權重。§1 / §2 任一為空時、回頭補完 Stage 1 / 2、不得進入 scaffold 建議或交付決策記錄。

## Scaffold 建議格式

決策記錄經使用者確認後、產出 scaffold 建議。建議先以清單形式呈現、使用者點頭才實際建檔 — scaffold 是決策的執行、不是新一輪決策。

每個建議檔案兩欄：產生什麼、承擔哪條決策或底線。範例：

| 檔案 / 設定                                                                 | 承擔的決策或底線                                                         |
| --------------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| per-domain module 目錄骨架（每個 domain 一個 module、公開面集中一個入口檔） | Domain map 的 OCP 邊界 — 內部面不被跨 domain import 靠目錄結構而不是默契 |
| 行為情境測試骨架（每條 Given / When / Then 一個 pending test）              | 操作清單的驗收情境直接轉測試、BDD 的「情境先於實作」落地                 |
| `.env.example` + `.gitignore`                                               | 底線：secret 不進 repo                                                   |
| `docker-compose.yml`（DB + app）                                            | 維度決策：managed PostgreSQL 的本地開發等價環境                          |
| `migrations/` + migration 工具設定                                          | 底線：schema 版本化                                                      |
| `.github/workflows/ci.yml`                                                  | 底線：CI gate（測試 + lint 過才 merge）                                  |
| `/healthz` endpoint stub                                                    | 底線：health check                                                       |
| log 設定（JSON + request id）                                               | 底線：structured log + 錯誤分類                                          |
| `README.md` 部署 / 回滾段                                                   | 底線：部署可回滾（回滾指令文件化）                                       |
| `docs/tech-decisions.md`                                                    | 決策記錄本體進 repo、跟程式碼同生命週期                                  |

兩個邊界：

- **Scaffold 範圍止於「決策的直接執行」**：業務邏輯、資料 model、UI 不在 scaffold 範圍；scaffold 完成時 repo 應該是「防護底線就位、第一個 feature 可以開始寫」的狀態。
- **決策改、scaffold 跟著重生**：使用者中途改決策（換 DB、換部署平台）時、回到決策記錄改對應段落、再重出受影響的 scaffold 項、不直接在舊 scaffold 上手改出漂移。

## 銜接 doc 系統（專案有 doc skill 時）

決策記錄是 saas 的終點、也是 doc 需求文件系統的起點 — 它已含 doc 需要的全部原料、銜接只做格式移交、不重新訪談。偵測 `.claude/skills/doc/` 存在時、依下表把決策記錄各段移交為 doc 文件。

**前置檢查（先過閘門再移交）**：移交前確認 §1 操作風險表、§2 Domain Map 非空（填寫規則 5）。任一為空＝訪談沒走完 Stage 1 / 2、此時 doc 的 usecase / spec 無源可長 — 回頭補盤點、不可硬生半成品文件。

| 決策記錄段落 | → doc 文件 | 映射 |
| --- | --- | --- |
| §1 操作風險表（BDD） | usecase（每個操作主體一個 UC） | 操作→用例、主情境→主成功場景、失敗情境→例外場景、風險 + 前端引導 + 後端防護→驗收條件 |
| §2 Domain Map | spec（每個自建 domain 一份）的 domain 邊界 + 責任 | domain→spec 目錄、責任→概述、command→FR（與 §3 雙源） |
| §3 技術維度決策 | spec 的 FR / NFR（與 §2 雙源）+ proposal 的技術決策 | 需求判讀→FR、選型 / 防護→NFR + proposal 決策依據 |
| §2 介面契約段 | spec 的資料模型 + 介面規格章節 | payload schema / 子協議 / 資料模型→spec 介面規格 |
| §0 定錨 + 交付形態 gate + §4 / 5 底線 / tripwire | proposal | 定錨 + gate→需求來源 / 範圍界定、防護底線→驗收條件、tripwire→Out-of-Scope |

spec 的 FR 是雙源：§2 Domain Map 給 domain 邊界與責任、§3 技術維度給技術性 FR / NFR — 兩段都要回看、缺一則 spec 不完整。

移交步驟：(1) 先生成 proposal（綁範圍）、(2) 再依 domain map 生成各 spec、(3) 再依操作表生成各 usecase、(4) 補雙向交叉引用（proposal 的 spec_refs / usecase_refs、spec 的 related_usecases、usecase 的 related_specs）、**(5) 推導標記審查**（見下段）、(6) CLAUDE.md 瘦身 — 需求文件結構化落地到 docs/ 後、CLAUDE.md 中的完整技術規格替換為路由索引表（只留決策編號 + 維度 + 選型一行摘要 + 指向 `docs/tech-decisions.md` 的路徑）、auto-load token 隨之下降。doc 端的接手細節見 doc skill 的「與 saas-tech-selection 的銜接」節。doc 產出物進入 TDD 流程時，由 `/tdd start` 自動偵測並執行銜接（見 `.claude/skills/tdd/references/doc-handoff.md`）。

### 推導標記審查（Step 5）

Spec 的介面規格（endpoint 路徑、參數格式、回應結構、匹配語法、資料模型 DDL）和實作策略（聚合方式、評估時機、排程頻率）通常不在訪談中逐一確認——訪談確認的是「需要什麼能力」，介面細節由 Claude 在移交時推導填補。推導的設計合理但未經用戶確認，可能和用戶心中的設計（或專案既有的設計文件）不一致。

移交完成後、交付用戶 review 前，執行一輪推導標記審查：

**掃描範圍**：所有 spec 的「介面規格」和「資料模型」章節。

**標記規則**：

| 來源 | 標記 | 說明 |
| --- | --- | --- |
| 訪談中用戶明確說過 | 不標記 | 有訪談原話可回溯 |
| 決策記錄 §2 介面契約有寫 | 不標記 | 有契約可回溯 |
| 既有設計文件（教學/RFC/ADR）有定義 | 不標記（但須比對一致性） | 有權威來源可回溯 |
| Claude 推導填補 | 在該項目後加 `<!-- inferred -->` 註記 | 提醒用戶此項需確認 |

**既有設計文件比對**：若專案有 CLAUDE.md 指向既有設計文件（教學文件、RFC、ADR），移交時必須讀取對應文件比對。推導結果與既有設計不一致時，以既有設計為準並在 spec 中引用來源。

**交付格式**：Spec 交付時附一份「推導項清單」摘要，列出所有 `<!-- inferred -->` 標記的項目，讓用戶一眼看到哪些需要確認。推導項清單是暫時性產物——用戶確認後移除 `<!-- inferred -->` 註記。

### 關鍵介面主動確認（Step 5 附屬）

推導標記審查中，以下類型的介面決策因影響範圍大（改動需同步多個消費端），應主動向用戶確認而非僅標記：

| 介面類型 | 確認內容 | 原因 |
| --- | --- | --- |
| HTTP endpoint 路徑 | 資源命名（`/events`）vs 操作命名（`/query`）、版本前綴（`/v1/`） | 路徑是 SDK 和文件的穩定錨點，改動成本高 |
| 資料模型 DDL | 表結構、索引策略、欄位拆分方式 | 影響查詢效能和 migration 路徑 |
| 聚合/保留策略 | 聚合摘要表 vs 原始事件保留、保留期限分層 | 影響儲存架構和查詢 API 設計 |
| 查詢參數語法 | 萬用字元（`*`）、正規表達式、前綴匹配的語意定義 | 影響 SDK 端的 API 使用方式 |

確認方式：用 AskUserQuestion 逐項列出推導的設計和替代方案，讓用戶選擇或提供自己的設計。

買掉（外包 vendor）的 domain：在 spec 標整合邊界、不展開內部 FR；其 reliability 第三方依賴訪談結論進 spec 的設計約束。
