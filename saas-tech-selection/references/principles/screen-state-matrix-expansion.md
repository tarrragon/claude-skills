# Screen State Matrix Expansion

> **角色**：本卡是 `saas-tech-selection` 的支撐型原則（principle）、被 `user-operations-bdd.md` 引用。
>
> **何時讀**：Stage 1 操作盤點完成後、進入 Stage 2 之前 — 操作清單的「前端引導」欄只描述顯示、容易漏掉操作和退出路徑。

## 原則

操作盤點產出的「前端引導」欄通常只描述「使用者看到什麼」（顯示），不描述「使用者能做什麼」（操作）和「使用者怎麼離開」（退出路徑）。畫面狀態矩陣把每個畫面的每個狀態展開成四欄（顯示 / 可用操作 / 進入條件 / 退出路徑），退出路徑欄為空代表 UX 死胡同。

## 展開步驟

1. 從操作清單抽出所有涉及的畫面
2. 每個畫面列出所有狀態（從 BDD 的 Given/Then 推導、加上程式碼中的狀態 enum）
3. 每個狀態填四欄 — 「顯示」從操作清單的前端引導欄直接填入，「操作」和「退出路徑」需要主動補
4. 退出路徑欄為空的狀態 = UX 死胡同，在實作前修正

## Gate fallback 同步

每個 gate（認證、網路、權限、環境、付費、版本相容性）的三問（成功時做什麼 / 失敗時做什麼 / 使用者不知道發生什麼時做什麼）應該在狀態矩陣中有對應的行。Gate 的「失敗」欄位為空 = 使用者被擋住無法繼續。

## 為什麼在 Stage 1.5 而非 Stage 2

畫面狀態矩陣是 BDD 操作盤點的下游（需要操作清單作為輸入）、但是 domain/event 切分的上游（狀態轉換事件可能成為 domain event）。放在 Stage 1 → Stage 2 之間、確保 UI 層的缺口在進入系統設計之前就被發現。
