# 錯誤恢復：訊息、重試與降級

本 reference 為「設計或審查錯誤發生後的使用者路徑」情境。

適用：撰寫錯誤訊息文案；設計重試機制（自動 / 手動、間隔策略）；使用者卡在「失敗 → 重試 → 失敗」迴圈；部分功能因外部依賴不可用時的降級呈現。
不適用：錯誤還沒發生時的關卡設計（gate 與 fallback 範疇）；錯誤狀態在整個畫面狀態機中的定位（狀態矩陣範疇）— 本文聚焦錯誤發生後的訊息、重試與降級三件事。

> **自包含聲明**：閱讀本文件不需要先讀其他 reference。

---

## 核心概念

錯誤不只是紅色文字 — 是一個需要設計退出路徑的狀態。錯誤恢復設計的三個組成：錯誤訊息讓使用者理解與決策、重試機制讓可恢復的失敗自動或手動恢復、降級模式讓局部失敗不擴大成整體不可用。三者共同的底線：使用者在任何錯誤狀態下都有至少兩條路（繼續嘗試、或離開）。

## 錯誤訊息的兩個職責

錯誤訊息承擔診斷（使用者能讀懂發生什麼）和行動（使用者知道下一步做什麼）。缺診斷讓使用者焦慮（「出了什麼事？」）、缺行動讓使用者卡住（「然後呢？」）。

**診斷面**：

- 用使用者的語言 —「無法連線到伺服器」優於 `ECONNREFUSED 127.0.0.1:7681`。技術細節保留在次要位置（折疊區塊、「詳細資訊」、複製按鈕），進階使用者能取得、一般使用者不被打擾。
- 描述影響而非原因 —「終端機暫時無法使用」比「WebSocket 連線逾時」直接回答「這對我意味著什麼」。原因是補充、影響是核心。
- 語氣反映實際嚴重程度 —「嚴重錯誤」「系統崩潰」類措辭在使用者只能重試或返回的情況下是恐嚇、不是資訊。

**行動面**：

- 每個錯誤訊息至少一個可執行的下一步，越具體越好 —「連線失敗，請檢查網路後重試」給了兩個行動；「發生錯誤」給了零個。
- 行動要對使用者可操作 —「請聯繫系統管理員」在自用工具場景沒有意義（使用者就是管理員）；「請稍後再試」在服務完全不可用時沒有幫助。
- 行動選項中至少一個是退出路徑（返回 / 關閉 / 取消）— 使用者可能不想重試、只想離開。
- 行動與錯誤層級對位 — 行動解決的層級等於錯誤發生的層級：資料層錯誤（檔案格式不合法）配資料層行動（換檔案重試）；重載 app / 重啟這類執行環境層行動只配執行環境層錯誤。過重（資料層錯誤配重載）不解決問題、還附帶狀態遺失的代價；過輕（執行環境層崩潰只給「關閉」）讓使用者無路可走 — 兩個方向都是缺口，而使用者傾向照著按鈕走。通用錯誤容器把最重的行動配給所有錯誤是不對位的結構性來源 — 錯誤 UI 的複用邊界是「行動相容」、不是「都是錯誤」。

**三層結構**：標題（一句話描述影響）、說明（補充原因與 context）、行動（按鈕或連結）。輕微錯誤只需標題 + 一個行動；嚴重錯誤三層都要。

## 重試機制

策略選擇取決於兩個因素：失敗是否可能自行恢復（暫時性網路中斷 vs 伺服器不存在）、使用者是否願意等待（前景操作 vs 背景同步）。

**自動 vs 手動 vs 混合**：

| 模式 | 適用                                     | UX 要求                                                     |
| ---- | ---------------------------------------- | ----------------------------------------------------------- |
| 自動 | 背景操作（同步、上報、心跳）、暫時性失敗 | 讓使用者知道正在重試（「第 2 次嘗試」）；必須有上限         |
| 手動 | 前景操作（使用者主動發起）、需確認意圖   | 重試按鈕明顯、旁邊有退出路徑                                |
| 混合 | 連線類操作的常見模式                     | 先自動 N 次（如 3 次），失敗後切換到「失敗」畫面 + 手動重試 |

自動重試無上限的問題：不可恢復的失敗（伺服器已關閉、認證過期）下浪費資源和電量、且使用者無法察覺問題。混合模式切換 UI 的意義是向使用者傳達「自動恢復已經試過了、需要你判斷接下來怎麼做」— 使用者可能知道原因（忘了開伺服器、WiFi 沒連上），修正後再手動重試。

**間隔策略**：

- 立即重試：極短暫瞬態失敗（DNS 偶發、TCP reset）。風險是 server 過載時多 client 同時重試的 thundering herd。
- 固定間隔：簡單可預測、使用者能估算等待。
- 指數退避 + jitter：間隔加倍（1s → 2s → 4s → 8s），適合 server 過載場景 — 給 server 恢復時間、減少 client 消耗。UX 挑戰是等待越來越長，解法是顯示倒數（「12 秒後自動重試」）加手動重試按鈕（使用者可跳過等待）。

**重試狀態的 UI**：使用者需要知道三件事 — 正在重試、已重試幾次、下一次在什麼時候：

```text
連線失敗，正在重新連線...
第 2 次嘗試（共 5 次上限）
下次重試：8 秒後 [立即重試] [返回首頁]
```

## Error loop 的逃生口

「失敗 → 重試 → 失敗」迴圈困住使用者的三個條件同時成立時發生：錯誤持續存在（不會因重試改變）、UI 只提供重試選項、沒有重試次數上限。典型形態：error 畫面上唯一的按鈕是「重連」，底層問題（伺服器關閉、認證過期）不會因重連消失，使用者唯一的出路是殺掉 app。

逃生口設計：

1. **每個 error 畫面至少兩個選項** —「重試」+「返回」是最小組合。
2. **自動重試達上限後切換 UI** — 從「正在重試」切到「失敗」+ 手動選項。
3. **提供診斷線索讓使用者判斷值不值得繼續重試** —「伺服器沒有回應」（可能正在重啟、重試或許成功）和「認證已過期」（重試不會改變、需要重新登入）是不同的決策。
4. **替代操作路徑** — 連特定伺服器失敗提供「選擇其他伺服器」、認證失敗提供「其他登入方式」，把「重試同樣的操作」擴展成「嘗試不同的操作」。

審查方法（設計階段可完成）：找出所有 error 類狀態，檢查可用操作欄是否只有「重試」、退出路徑欄是否為空 — 兩者同時成立 = 潛在 retry loop。

## Degraded mode

部分功能因外部依賴不可用（斷網、服務故障、權限缺失）而暫停時，讓使用者清楚知道哪些可用哪些不可用，而非整個 app 停擺。三種呈現方式：

- **靜默隱藏**：把不可用功能從 UI 移除。只適合使用者從未用過的功能 — 用過的功能突然消失會被當成 bug、恢復後重新出現加劇困惑。
- **明確標示**：UI 保留、加不可用標示。要點：說明原因（「搜尋功能需要網路連線」優於單純灰掉）、說明恢復條件（「連上網路後自動恢復」）、不只靠顏色（搭配文字或圖示）。
- **替代方案**：提供不需要失敗依賴的版本（線上搜尋 → 本地快取搜尋），並讓使用者知道在用替代版（「離線模式 — 結果可能不是最新」）。

**全域 vs 功能級**：全域降級（頂部橫幅 + 所有網路功能統一標示）實作簡單、適合網路是核心依賴的 app，但可能過度限制；功能級降級（每個功能獨立宣告依賴、獨立判斷可用性）更精確、實作更複雜。

**進入與退出**：

- 進入用持續顯示的通知（橫幅類）而非自動消失的短提示 — 短提示消失後使用者忘記自己在降級中、之後的操作失敗顯得毫無脈絡。
- 防抖動：網路不穩時每秒在正常 / 降級間切換會讓 UI 閃爍，加穩定性判斷（連續 N 秒不可用才進、連續 N 秒可用才出）。
- 退出時通知（「已恢復連線」）並自動執行待完成操作（同步、補發）。

## 檢查清單

- [ ] 每個錯誤訊息都有診斷（使用者語言、描述影響）和行動（至少一個可執行下一步）？
- [ ] 行動選項中有退出路徑？
- [ ] 每個錯誤 UI 的行動層級與錯誤層級對位（過重與過輕都是缺口：無資料層錯誤配重載、無環境層錯誤只給關閉）？
- [ ] 技術細節收在次要位置而非主文案？
- [ ] 自動重試有上限、達上限後切換到手動模式？
- [ ] 重試中的 UI 顯示次數與下次時間？
- [ ] 指數退避場景有倒數顯示 + 手動跳過？
- [ ] 沒有「可用操作只有重試 + 退出路徑為空」的狀態？
- [ ] 降級狀態用持續通知呈現、說明原因與恢復條件？
- [ ] 降級的進入退出有防抖動？
