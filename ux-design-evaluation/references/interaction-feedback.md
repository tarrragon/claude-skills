# 互動回饋：點擊確認、等待指示、結果通知

本 reference 為「設計或審查使用者動作的系統回應」情境。

適用：審查按鈕回饋完整性；判斷 loading indicator 的時機與形式（spinner / skeleton / 進度條）；設計非同步操作的按鈕生命週期與防重複提交；連線 / 配對 / 同步類多步驟流程的畫面級回饋；操作結果的通知形式選擇（SnackBar / Dialog / Banner / Bottom Sheet）；元件語意與版面（切換標籤、可點性辨識、選中態對比、溢出 affordance、回饋文字的版面保障）。
不適用：畫面狀態機的完整設計（狀態矩陣範疇 — 本文的畫面級回饋聚焦「每個狀態給什麼指示」，狀態的枚舉與退出路徑歸狀態矩陣）；錯誤訊息的文案內容（錯誤恢復範疇）。

> **自包含聲明**：閱讀本文件不需要先讀其他 reference。

---

## 核心原則

**每個使用者動作都需要即時、誠實的系統回應。** 沒有回饋的按鈕等同壞掉的按鈕 — 使用者無法區分「系統在處理」和「按鈕壞了」。對應 Nielsen 十大可用性啟發法第一條（系統狀態可見性）。「誠實」的含義：指示與實際狀態一致 — 假進度條、與實際無關的百分比，被識破後信任一次性崩壞。

## 三層回饋模型

使用者按下按鈕後，回饋依時間順序分層：

| 層          | 時間點    | 回答           | 實作                            |
| ----------- | --------- | -------------- | ------------------------------- |
| 1. 點擊確認 | 0-100ms   | 「收到操作了」 | 視覺狀態變化 / ripple / 觸覺    |
| 2. 等待指示 | 100ms-10s | 「正在處理」   | spinner / loading 狀態 / 進度條 |
| 3. 結果通知 | 操作完成  | 「結果是什麼」 | 畫面更新 / SnackBar / 錯誤訊息  |

第一層必須在 100ms 內出現 — 平台原生元件的 pressed / ripple 互動態已滿足，自建元件（自繪按鈕、canvas UI）才需要自行驗證延遲。第三層的四種結果都要設計：成功（讓使用者看到改變）、失敗（原因 + 下一步）、空結果（區別「沒找到」和「沒載入」— 明確文字 + 下一步建議）、部分成功（摘要先告知整體、明細列失敗項與修正動作，讓使用者只處理失敗部分）。

**結果通知的鏈路前提**：第三層假設「結果會正確到達 UI」。UI 與執行端分屬不同 context 時（browser extension 的 popup / content script / service worker、跨 process 架構），結果要跨訊息通道才到 UI — 通道語意錯誤（async listener 在共享通道上把 undefined 搶先當回應送回）或事件訂閱缺失（handler 沒注入、事件沒人接）會讓通知消失、甚至反轉成「操作成功、UI 報失敗」— 比零回饋更糟，使用者會重做操作並失去對介面的信任。跨 context 的結果通知要有端對端驗證（操作成功 → UI 顯示成功），不只測 UI 收到資料後的呈現。

## 時間門檻與回饋策略

回饋形式取決於等待時間 — 100ms 的頁面切換加 spinner 是視覺雜訊、30 秒的批次匯出只有 spinner 撐不住耐心。門檻來源：100ms / 1s / 10s 是感知與注意力門檻（Miller、Nielsen）、400ms 是生產力門檻（Doherty Threshold）：

| 等待時間  | 使用者感受 | 回饋策略                         | 範例               |
| --------- | ---------- | -------------------------------- | ------------------ |
| 0-100ms   | 即時       | 點擊確認即可                     | 頁面導航、開關切換 |
| 100-400ms | 幾乎即時   | 動畫過渡掩蓋等待                 | 快速搜尋、本地查詢 |
| 400ms-1s  | 短暫等待   | Spinner / 按鈕 loading 狀態      | API 查詢、表單送出 |
| 1-10s     | 明顯等待   | 進度指示 + spinner               | 檔案上傳、資料匯入 |
| > 10s     | 長時間等待 | 進度百分比 + 預估時間 + 取消選項 | 批次處理、大型同步 |

超過 10 秒使用者開始考慮放棄；數十秒以上考慮轉背景任務、完成後通知。

**完成宣告要有窮盡證據**。批次 / 遍歷類操作的「完成」判定若只憑停滯訊號（連續 N 輪計數沒有變化），分不出「真的到底了」和「載入通道根本沒打開」— lazy-load 頁面的內容增長依賴特定觸發動作，動作缺失時計數永遠穩定，實測出現過只處理十分之一就宣告完成的事故。窮盡證據包括：來源總數對照（頁面計數、API total）、終止標記（「沒有更多」元素、載入觸發器消失）、API 側的窮盡宣告（分頁 cursor 耗盡、has_more=false）。只有停滯訊號時，完成宣告降級為「已取得 N 筆、可能還有更多」的部分結果表達；下游需要全量的操作（備份、匯出、合規留存）不適用降級 — 不完整按失敗處理、阻止下游動作。

**延遲是分布、不是常數**。同一支 API 平常 300ms、尖峰或弱網下 3s，對表用 p90 / p95 而非平均值 — 平均 300ms、p95 卻 2s 的操作，每二十次就有一次使用者盯著沒有指示的畫面等 2 秒。兩個補充機制：

- **延遲顯示（delayed spinner）**：發請求後等 300-400ms 才掛 spinner、請求先回來就完全不顯示 — 快的請求（多數）沒有閃爍、慢的請求在使用者開始懷疑前得到指示。這是延遲分布橫跨門檻帶時的標準解。
- **最短顯示時間**：spinner 一旦顯示至少停留約 300ms、請求提早完成也不立即移除 — 消除「閃一下就消失」的視覺雜訊。

兩個數值都是業界慣例值、可依實際延遲分布調整。驗證方式：用網速節流（DevTools / proxy 限速）模擬慢回應、逐帶確認對應的 UI 確實出現。

## 按鈕級回饋

**非同步按鈕**（API / DB / 權限請求 — 點擊後需等待外部回應）：

```text
[idle] → 點擊 → [loading: disabled + spinner] → 完成 → [idle + 結果通知]
```

- loading 期間 disabled（防重複提交 — UI 層降低發生率、最終防線是伺服器端冪等，兩層都要做）
- 按鈕文字可變（「送出」→「送出中...」）
- 結束後必須恢復 idle — 不論成功失敗；配 timeout 兜底（取值略大於 client / 後端逾時，逾時後恢復 + 錯誤訊息）
- 結果通知獨立於按鈕（SnackBar 或畫面更新），不塞在按鈕裡

**同步按鈕**（導航 / 切換 / 開關 — 本地立即完成）：

```text
[idle] → 點擊 → [視覺回饋 + 執行] → [完成]
```

- 不需要 loading 狀態
- 防連點用 leading-edge debounce（第一次點擊立即執行、之後約 300ms 內忽略）或導航鎖（in-flight flag）。trailing debounce（等 300ms 沒新點擊才執行）會給每次操作加 300ms 延遲、違反 100ms 門檻 — 防的是第二次點擊、第一次要立即生效。不用 disabled 防連點（會讓按鈕每次點擊閃跳灰態）。

**六種按鈕狀態** — 每種狀態傳達一種系統訊息、缺少任何一種使用者就失去對應判斷依據：

| 狀態     | 傳達訊息   | 設計要點                                                                                  |
| -------- | ---------- | ----------------------------------------------------------------------------------------- |
| Default  | 可以按     | 一眼可辨識為按鈕                                                                          |
| Hover    | 可互動元素 | 僅桌面端（行動裝置無 hover）                                                              |
| Active   | 收到點擊   | 對應第一層點擊確認；視覺差異要夠大、可配觸覺                                              |
| Focus    | 焦點在此   | 鍵盤導航必要（WCAG 要求可見焦點指示）                                                     |
| Disabled | 現在不能按 | 說明原因（常駐文字優先 — tooltip 對行動端與鍵盤使用者不可達）；條件與使用者無關時考慮隱藏 |
| Loading  | 正在處理   | 自動帶 disabled、有 timeout、結束必恢復                                                   |

實作落點在共用 Button 元件層、不是逐顆按鈕 — 平台元件庫已內建前四種，通常只需補 Loading / Disabled 的語意。估工按「元件層一次工 + 各按鈕接狀態來源」，不是「按鈕數 × 狀態數」（後者高估數倍、讓完整狀態設計看起來做不起）。

**元件語意與版面檢查** — 狀態清單之外，標籤、圖示、選中態與版面各自承載語意：

1. 切換元件的標籤要能區分「現態」與「動作」— 名詞標籤（「管理模式」）容易被讀成當前狀態指示；消歧做法是狀態顯示與動作按鈕分離（非互動現態文字 +「切換」動作鈕）、改用自帶 on/off 語意的控件（switch / segmented control）、或最低成本把標籤改成動詞句（「切換到 X」— 不動結構、現態仍不可見）。不幫助消歧的圖示是雜訊、移除。
2. 非互動的狀態指示不與動作按鈕同形混排 — 同列有按鈕時使用者預設整列可點，點到指示的「沒反應」被讀成壞掉；描邊 vs 實心不是可靠的互動性訊號，指示用明顯非按鈕形態（文字 chip、圓點）或移出動作列，同一圖形全畫面只承載一個語意。
3. 選中態是（底色, 文字色）的成對設計 — 單獨覆寫選中底色會脫離主題的成對機制、元件庫不會為自訂底色反推文字色，淺底疊淺字就是「選中後看不到字」；對比方向成對反轉（底淺配深字），逐狀態驗 WCAG AA（一般字級 4.5:1、大字 3:1）。
4. 水平清單溢出要有捲動 affordance（邊緣漸層、下一項半顆露出、箭頭、捲軸指示）— 截斷的預設讀法是「壞了 / 被旁邊的元件遮住」、不是「可以捲」；選項少時優先消除溢出（wrap / 選單），截斷邊緣避免緊貼固定按鈕（視覺歸因會錯）。
5. 關鍵回饋文字要有版面保障 — flex 寬度競爭會把「已選擇: 3/45」壓成「...」（state 正確、使用者零資訊）；ellipsis 對短關鍵資訊是錯誤的降級策略，改縮格式或最小寬度，回饋鏈要驗到像素層、並在窄幕驗收。
6. 版面擠壓先分「換算錯 vs 分配錯」再動手 — 等比縮放（換算類）sizing 套件不驗證空間分配：換算工具工作在常數層（設計稿座標 → 裝置座標、發生在 layout 之前）、空間分配發生在 layout 協商層（有限寬度分給動態內容：計數、i18n 字串、使用者資料），兩層獨立、比例正確不等於空間夠用（隱含假設「實際內容 = 設計稿內容」被動態內容打破）。換算錯查套件用法；分配錯補三層（設計層空間競爭規格：誰讓步怎麼降級；實作層把 flex 組合當顯式優先權決策；驗證層窄幕驗收 / layout test）。其他響應式方案（constraint-based layout、breakpoint）保證層各異；引入「解決某維度」的套件時，記錄它不保證的層、缺口層掛對應驗證。

## 畫面級回饋

按鈕級處理單一操作；連線、配對、同步、多步驟表單涉及**整個畫面的狀態轉換**，每個中間狀態需要能被區辨的 UI 與退出路徑：

| 維度     | 按鈕級                       | 畫面級                                       |
| -------- | ---------------------------- | -------------------------------------------- |
| 範圍     | 單一按鈕內的狀態變化         | 整個畫面 UI 替換                             |
| 狀態數   | 2-3（idle / loading / 結果） | 4-6（idle / 處理中 / 成功 / 失敗 / 斷線...） |
| 退出路徑 | 不需要（按鈕自己恢復）       | 每個中間狀態都必須有                         |

以連線流程為例的狀態回饋：

| 狀態         | UI 呈現                  | 可用操作    | 退出路徑                        |
| ------------ | ------------------------ | ----------- | ------------------------------- |
| idle         | 連線按鈕                 | 點擊連線    | 返回上一頁                      |
| connecting   | 全頁 spinner +「連線中」 | 取消        | 取消 → idle                     |
| connected    | 功能畫面                 | 使用 / 斷開 | 斷開 → idle / 返回              |
| error        | 錯誤訊息 + 重連按鈕      | 重連 / 返回 | 重連 → connecting / 返回 → idle |
| disconnected | 「連線中斷」+ 重連按鈕   | 重連 / 返回 | 重連 → connecting / 返回 → idle |

關鍵設計：error 和 disconnected 必須同時提供「重連」和「返回」— 只有重連會把使用者鎖在錯誤迴圈。timeout 是 connecting 的隱形退出路徑：取消靠使用者主動、timeout 是系統兜底（取值略大於 client / 後端逾時、讓真正的失敗先回來），逾時後轉入 error。

## Spinner vs Skeleton

| 條件                                   | Spinner              | Skeleton Screen                  |
| -------------------------------------- | -------------------- | -------------------------------- |
| 結果形狀已知（列表 / 卡片 / 個人資料） |                      | 適合：佔位接近最終結果、降低焦慮 |
| 結果形狀不確定（連線 / 計算結果）      | 適合：無法預畫佔位   |                                  |
| 操作觸發的等待（按鈕後）               | 適合：按鈕內 spinner |                                  |
| 頁面首次載入                           |                      | 適合：給畫面結構預期             |
| 等待短且可預測（< 2s）                 | 適合：輕量指示       |                                  |
| 等待較長（2-10s）                      |                      | 適合：結構佔位優於空白等待       |

判斷式：「知道長什麼樣 → skeleton；不知道 → spinner」。條件同時命中不同建議時裁決順序是**形狀 > 觸發 > 時長** — 按鈕載入個人資料 1.5s，形狀已知就用 skeleton。Skeleton 的佔位尺寸要與最終內容一致 — 載入完成瞬間的版面跳動比空白等待更傷體驗。

其他感知效能技巧（真實效能無法改變時降低等待感受）：**樂觀更新**（先更新 UI、失敗再回滾 — 適合高成功率操作；回滾必須走失敗通知「操作未成功、已還原」+ 重試，不能無聲消失）、**漸進式載入**（先文字後圖片，等待期間可開始閱讀）、**動畫過渡**（300ms fade-in 掩蓋 200ms 延遲，感知是「動畫」而非「等待」）。

## 結果通知的形式選擇

通知形式是設計決策，判準是兩個軸的交叉：**是否需要使用者操作**（純告知可自動消失；需要回應必須持續顯示 — 自動消失等於系統擅自幫使用者選了「忽略」）× **干擾程度**（視覺佔用 + 互動阻斷，原則是剛好夠用）。

六種形式從低干擾到高干擾：

| 形式            | 干擾 | 特性與適用                                                                                     |
| --------------- | ---- | ---------------------------------------------------------------------------------------------- |
| Inline 狀態更新 | 零   | 元素原位更新（收藏填滿、欄位紅字）；結果在觸發元素的 context 內能完整呈現時的首選形式          |
| SnackBar        | 低   | 底部短暫訊息條、自動消失、可帶一顆 action；已完成的確認（「已儲存」）、可選操作（刪除 + 復原） |
| Banner          | 中   | 內容區頂部持續顯示；**狀態驅動**（「目前離線」）— 與事件驅動的 SnackBar 的關鍵差異是生命週期   |
| Bottom Sheet    | 中高 | 底部滑出、延伸當前操作（分享、篩選、選項多的選擇）；易被滑掉 dismiss、不承載重要決策           |
| Dialog          | 高   | 居中 modal、阻斷背景互動；不可逆操作確認、含關鍵資訊的結果、必須二選一的決策                   |
| 全螢幕          | 最高 | 批次結果明細（87 成功 / 13 失敗逐筆）、流程終結頁、需在結果上進一步操作                        |

選擇矩陣：

| 情境                         | 需要操作？ | 推薦形式          |
| ---------------------------- | ---------- | ----------------- |
| 操作成功、結果已反映在 UI    | 否         | Inline            |
| 操作成功、結果不在視線範圍   | 否         | SnackBar          |
| 操作成功、帶可選復原         | 可選       | SnackBar + action |
| 持續性狀態變更（離線、降級） | 否 / 可選  | Banner            |
| 結果需要選後續操作           | 是         | Bottom Sheet      |
| 不可逆操作的事前確認         | 是         | Dialog            |
| 結果含關鍵資訊（檔案路徑）   | 是         | Dialog            |
| 結果密度高、需閱讀與逐筆操作 | 是         | 全螢幕            |

**補充判準 — 操作頻率**：二軸無法區分「刪除帳號」和「刪除待辦事項」（讀數相同）。低頻高代價用 Dialog 確認、高頻低代價用 SnackBar + 復原。頻繁操作每次都彈 Dialog，使用者訓練出「不讀直接按確定」、真正重要的確認也被跳過 — Dialog 的保護效果與使用頻率成反比。

**自動消失時間**：純文字 SnackBar 約 4 秒、帶 action 8-10 秒（要讀完訊息再決定是否按）。無障礙：帶 action 的自動消失通知，action 要在別處可復得（如復原可從歷史觸發）、或螢幕閱讀器啟用時延長停留。Banner 與 Dialog 不自動消失 — Banner 在觸發條件消失時移除、Dialog 在使用者回應後關閉。

**堆疊規則**：SnackBar 新舊替換 — 批次操作逐筆彈只剩最後一條，應彙整成摘要或改全螢幕明細。Dialog 之上再彈 Dialog 是嚴重問題（注意力分裂、dismiss 順序混亂），連續確認改成多步驟單一 Dialog 或全螢幕流程。Banner 垂直堆疊、超過兩條考慮合併摘要。可接受的跨形式堆疊：Bottom Sheet 之上彈 Dialog（context + 確認、語意分工清楚）。

## 反模式速查

| 反模式                           | 使用者後果                     | 修正                                   |
| -------------------------------- | ------------------------------ | -------------------------------------- |
| 按鈕無視覺回饋                   | 重複點擊、重複觸發 callback    | 補第一層點擊確認                       |
| 非同步不禁用按鈕                 | 重複提交（重複扣款 / 寫入）    | loading 自動帶 disabled + 後端冪等     |
| loading 結束不恢復按鈕           | 以為介面當掉                   | 完成 / 失敗 / 逾時三個出口都恢復       |
| 只有 loading 無結果通知          | 不確定是否成功                 | 補第三層                               |
| 同步按鈕無防連點                 | 導航堆疊混亂（push 多次）      | leading debounce 或導航鎖              |
| 每個操作都全螢幕 loading         | 打斷心流                       | 指示只覆蓋被等待的區域                 |
| spinner 閃爍                     | 視覺雜訊                       | 延遲顯示 + 最短顯示時間                |
| 假進度 / 0% 跳 100%              | 信任崩壞                       | 步驟計數或 indeterminate 動畫誠實表達  |
| 畫面中間狀態無退出路徑           | 卡在 connecting、只能殺 app    | 每個中間狀態補退出（取消 / 返回）      |
| 錯誤狀態只有重連沒有返回         | 鎖在錯誤迴圈                   | 重試 + 返回兩條路                      |
| 所有通知都用 Dialog              | 訓練出「不讀直接按確定」       | 依二軸降級                             |
| 需要操作的通知用 SnackBar        | 沒讀完就消失                   | 改 Dialog 或 action 可復得             |
| 持續狀態（離線）用 SnackBar      | 消失後使用者忘記、後續操作困惑 | 改 Banner                              |
| 成功誤報失敗（通知鏈路錯誤）     | 重做操作、資料重複、信任崩壞   | 端對端驗證 + 通道上明確宣告回應權      |
| 完成宣告只憑停滯訊號             | 帶著殘缺資料離開而不自知       | 窮盡證據、或降級為部分結果表達         |
| 切換鈕的名詞標籤被讀成現態       | 誤判目前所在模式               | 拆成狀態顯示 + 動作按鈕、或 switch     |
| 狀態圖示與動作按鈕同形混排       | 點指示沒反應、被讀成壞掉       | 指示改非按鈕形態、移出動作列           |
| 關鍵回饋文字被版面壓成省略號     | 回饋存在、使用者拿到零資訊     | 最小寬度保障、縮格式優於 ellipsis      |
| 佔位 handler 上線（toast / log） | 可點的假按鈕、被讀成壞掉       | 隱藏或 disabled + 說明、release 前掃描 |

## 檢查清單

按鈕級：

- [ ] 點擊有視覺回饋（100ms 內）？
- [ ] 非同步：loading + disabled + 完成恢復 + timeout？
- [ ] 結果有通知（成功 / 失敗 / 空 / 部分成功）？
- [ ] 同步按鈕有 leading-edge 防連點？

畫面級：

- [ ] 每個中間狀態能被區辨、有退出路徑？
- [ ] 等待狀態可取消、有 timeout 兜底？
- [ ] 錯誤 / 中斷同時提供重試和返回？

等待與通知：

- [ ] 對表用 p90 / p95 而非平均延遲？
- [ ] spinner 有延遲顯示 + 最短顯示時間？
- [ ] spinner vs skeleton 按形狀 > 觸發 > 時長裁決？
- [ ] 通知形式對照二軸 + 操作頻率？
- [ ] 批次操作結果有彙整、無 SnackBar 成串 / Dialog 堆疊？
- [ ] 跨 context（extension / 多 process）的結果通知有端對端驗證？
- [ ] 批次 / 遍歷操作的完成宣告有窮盡證據（總數對照 / 終止標記）？

語意與版面：

- [ ] 切換型元件的標籤是動作語意（或已拆成狀態顯示 + 動作按鈕）？
- [ ] 非互動指示與動作按鈕形態可區分、圖示無撞名？
- [ ] Selected 態底色與文字色成對設計、對比達 WCAG AA？
- [ ] 水平溢出清單有捲動 affordance（或已消除溢出）？
- [ ] 關鍵狀態文字有版面保障（窄幕不被壓成省略號）？
- [ ] 版面擠壓的診斷分過層（換算錯 vs 分配錯）、沒把 sizing 套件當空間分配的保證？
- [ ] 無佔位 handler（dev toast / log-only）殘留（統一佔位標記 + release 前 grep 掃描）、未接線功能已隱藏或 disabled + 說明？
