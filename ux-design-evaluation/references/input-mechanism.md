# 輸入機制：鍵盤、送出、IME 的設計決策

本 reference 為「設計或審查使用者輸入介面」情境。

適用：任何有輸入框的功能（表單、搜尋、CLI / terminal、聊天）；mobile 鍵盤行為異常（自動校正破壞輸入、建議列遮擋畫面）；密碼 / API key / secret 類欄位的安全審查；送出時機的選擇（按送出 / 即時 / debounce）。
不適用：輸入內容的業務驗證邏輯（屬後端 / domain 層）；送出按鈕本身的 loading 回饋（屬互動回饋範疇）。

> **自包含聲明**：閱讀本文件不需要先讀其他 reference。

## 核心概念

輸入機制是設計產物，在功能規格階段決定，和 API schema 同級。手機鍵盤的行為由多個參數控制，每個參數都是設計決策 — 影響使用者體驗、UI layout、甚至通訊協議。依賴預設值等於把決策交給平台、事後逐一踩到問題再補。

## 四個決策維度

每個輸入框過一次四維決策，結果記錄成表、code review 時逐項對照：

**Keyboard type — 顯示哪種鍵盤**。判斷依據是「使用者要輸入什麼內容」：email 用 email 鍵盤（有 `@` 鍵）、電話用數字鍵盤、密碼或 CLI 指令用 `visiblePassword` 型別（避免自動校正和建議）。CLI 指令包含路徑分隔符、flag 縮寫等非自然語言內容，一般文字鍵盤會嘗試把 `ls -la` 或 `/usr/bin/` 校正成其他東西。

**Submit model — 怎麼送出**。整行送出（按 Enter / Send 一次傳整行）vs 逐字元送出（每個按鍵即時傳送）。這個決策直接影響通訊協議：整行送出的 server 按行處理、protocol 簡單，代價是 Tab 補全等即時互動無法觸發；逐字元送出支援補全和即時控制鍵、但 protocol 複雜度和網路延遲敏感度顯著提高。決策應在 protocol spec 階段做、不是 UI 實作時臨時決定。

**IME policy — 輸入法的自動行為**。三個控制項各自獨立：

- 自動校正（autocorrect）：把輸入替換成字典詞。非自然語言內容（指令、路徑、代碼）會被破壞。
- 輸入建議（suggestions）：鍵盤上方的候選列。可能遮擋畫面關鍵區域。
- 個人化學習（personalized learning）：IME 從輸入學習新詞、跨 app 適用。輸入可能含 secret 時是安全問題（見 IME 安全清單）。

**Special keys — 特殊按鍵**。手機鍵盤沒有 Esc / Tab / Ctrl / 方向鍵，需要這些按鍵的應用必須自建 UI（如鍵盤上方的工具列）。工具列的按鈕大小（可精確觸碰的最小尺寸約 44x44 pt）、排列順序、長按行為都是 UX 決策。

決策表範例（CLI 輸入場景）：

| 維度         | 選項            | 理由                         |
| ------------ | --------------- | ---------------------------- |
| Keyboard     | visiblePassword | CLI 指令不適用自動校正       |
| Submit       | 整行送出        | protocol 簡單，犧牲 Tab 補全 |
| IME          | 全關            | 安全考量 + 非自然語言輸入    |
| Special keys | 底部工具列      | 手機無實體 Esc/Tab/Ctrl      |

事先決策並記錄、review 逐項勾選，成本低於事後逐一發現問題 — 實例：一個終端機 app 的六個輸入參數全是實機測試後 hotfix 補上的，漏掉個人化學習關閉是安全漏洞、漏掉自動校正關閉是 UX 問題。

## 場景一：表單

表單輸入以自然語言為主（姓名、email、地址），鍵盤的自動行為大部分是幫助 — 和 CLI 場景的取捨方向相反。

**Keyboard type 按欄位選**：

| 欄位     | Keyboard type          | 理由                      |
| -------- | ---------------------- | ------------------------- |
| Email    | emailAddress           | 有 `@` 和 `.` 快捷鍵      |
| 電話     | phone                  | 只顯示數字和 `+` `-`      |
| URL      | url                    | 有 `.com` 快捷鍵和 `/` 鍵 |
| 密碼     | visiblePassword        | 關閉自動校正              |
| 數字金額 | number（含小數點選項） | 數字鍵盤加小數點          |

**驗證時機**：格式明確的欄位用即時驗證但在失去焦點（on blur）才顯示錯誤 — 輸入過程中報「email 格式錯誤」（使用者才打到 `user@`）是過早報錯；多欄位交叉驗證（密碼確認、日期範圍）用送出時驗證、配 scroll 定位 + 欄位 highlight 幫使用者找到錯誤欄位。混合策略：格式驗證 on blur、業務驗證（「email 已被註冊」）on submit。

**Auto-fill**：正確標記欄位語意類型（autofill hints），系統才能自動填入已儲存的姓名 / 地址 / 信用卡 — mobile 上減少的打字量直接轉化為轉換率。

**錯誤回饋**：訊息緊跟對應欄位下方（不集中在表單頂部）；內容說「期望什麼」而非「哪裡錯了」（「請輸入有效的 email 地址」優於「email 格式無效」）；錯誤欄位邊框變色搭配文字、不只靠顏色（色盲使用者）。

## 場景二：搜尋

核心決策是「輸入到什麼程度觸發搜尋」— submit model 維度在搜尋場景的展開：

| 觸發模式 | 機制                          | 適用                                             |
| -------- | ----------------------------- | ------------------------------------------------ |
| 按送出   | 打完按搜尋鈕、一次搜尋        | 搜尋成本高：DB 全文搜尋、外部 API（限速 / 計費） |
| 即時     | 每個字元觸發一次              | 搜尋成本低：本地過濾、記憶體內資料集             |
| Debounce | 停止輸入 300-500ms 後觸發一次 | 遠端搜尋的常見選擇：平衡即時感和 API 呼叫次數    |

即時觸發用在高成本搜尋的問題：輸入 `hello` 過程觸發五次 API 呼叫、前四次結果沒被看到就被覆蓋。Debounce 延遲的 trade-off：太短（100ms）接近即時觸發、太長（1000ms）使用者感到明顯延遲；搜尋框用 trailing-edge 語意（等停止輸入才觸發）。

結果顯示形式：suggestion list（候選可列舉：城市、產品、使用者名）、獨立結果頁（結果量大、需分頁）、即時過濾（在已顯示的清單中隱藏不符項目）。

細節：搜尋框 keyboard type 用一般文字 + `textInputAction: search` 讓 Enter 鍵顯示放大鏡 — 使用者看到搜尋圖示知道按下去會觸發搜尋、看到換行圖示會猶豫。

## 場景三：CLI / Terminal

CLI 指令是結構化語法 — 路徑分隔符、flag、管線符號都有精確語意，為自然語言設計的鍵盤自動行為（校正、建議、學習）全部變成干擾。桌面終端機直接傳按鍵事件、沒有 IME 層；手機在使用者和 app 之間插入 IME — 按鍵可能被組合、被校正、被學習。

處理：關閉全部 IME 自動行為（`visiblePassword` + autocorrect off + suggestions off + personalized learning off 四參數）；submit model 在 protocol 層決定（整行 vs 逐字元，影響 server 端行為預期和 frame 格式）；特殊鍵用鍵盤上方工具列提供（Esc / Tab / Ctrl / 方向鍵），Ctrl 組合鍵常見實作是「按 Ctrl 進入修飾模式、下一鍵自動加前綴」。

## IME 安全清單

IME 個人化學習把輸入存進跨 app 共用的詞庫 — 處理 secret 的輸入框若不關閉，密碼 / API key / 伺服器路徑會在使用者於其他 app 打字時出現在建議列，任何看到螢幕的人都看得到。這和傳統密碼外洩（資料庫入侵、傳輸攔截）路徑不同：使用者在日常打字時被動暴露、不會意識到 IME 記住了別的 app 輸入的密碼。

適用欄位：密碼 / PIN、API key / token、伺服器位址 / 連線字串、CLI 指令（可能含前述任何一類）、信用卡號、任何 confidential 欄位。

| 控制項     | 設定                     | 理由                       |
| ---------- | ------------------------ | -------------------------- |
| 個人化學習 | 明確關閉（不依賴預設值） | 防止 secret 進入 IME 詞庫  |
| 自動校正   | 關閉                     | 防止 secret 被替換成字典詞 |
| 輸入建議   | 關閉                     | 防止 secret 出現在建議列   |
| 密碼欄位   | 依需求設定 obscure text  | 畫面不顯示明文             |

平台邊界：這些設定對應 iOS 的 spell checking attribute 與 Android 的 no-suggestions flag，不同 IME app 的遵守程度不一 — app 端控制是「盡力而為」，設定正確的 flag 是必要條件、不是充分保證，安全敏感場景另配 secure text entry。

## 檢查清單

- [ ] 每個輸入框的四維決策（keyboard / submit / IME / special keys）記錄在規格？
- [ ] Keyboard type 對應欄位內容（不是全部用預設 text）？
- [ ] 非自然語言輸入（指令、代碼、ID）關閉自動校正？
- [ ] Secret 類欄位通過 IME 安全清單全部四項？
- [ ] Submit model 影響 protocol 的場景，決策在 protocol spec 階段做？
- [ ] 表單驗證時機分層（格式 on blur、業務 on submit）？
- [ ] 遠端搜尋用 debounce 且延遲在 300-500ms 區間？
- [ ] 需要特殊鍵的場景有自建 UI、按鈕尺寸可精確觸碰？
