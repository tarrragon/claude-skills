# 畫面狀態矩陣：狀態 × 操作 × 退出的完整性檢查

本 reference 為「設計或審查多狀態畫面」情境。

適用：設計連線 / 配對 / 同步這類多狀態流程的畫面；審查既有畫面的導航缺口；使用者反映「進了某個畫面出不去」；從 BDD 操作盤點展開 UI 設計；檢查路由定義與 UI 入口的一致性。
不適用：單一狀態的靜態頁面（沒有狀態轉換就不需要矩陣）；按鈕內部的 loading 狀態設計（屬互動回饋範疇）。

> **自包含聲明**：閱讀本文件不需要先讀其他 reference。

---

## 核心概念

畫面狀態矩陣是一張表格：每行代表一個畫面的一個狀態，每列描述該狀態的四個面向 — 顯示了什麼、使用者能做什麼、什麼條件進入、怎麼離開。目的是在實作前暴露導航缺口：**退出路徑欄位為空 = UX 死胡同**，使用者一旦進入就無法靠自己的操作離開，只能殺掉 app 或等系統逾時。

矩陣同時是可測試的設計產物 — 每一行可以直接轉成一條 UI test case（進入該狀態、驗證顯示與操作、執行退出路徑、斷言導航結果）。

## 四欄定義

| 欄       | 內容                                                       | 常見遺漏                                    |
| -------- | ---------------------------------------------------------- | ------------------------------------------- |
| 顯示     | 使用者看到的畫面元素：進度指示、錯誤訊息、表單、列表       | 少遺漏 — 規格通常都寫顯示                   |
| 可用操作 | 使用者在這個狀態下能執行的動作：按鈕、手勢、鍵盤輸入       | 常遺漏 — 規格寫「看到什麼」不寫「能做什麼」 |
| 進入條件 | 什麼事件讓畫面進入這個狀態：使用者操作、系統事件、外部事件 | 偶爾遺漏隱式轉換（自動觸發、timeout）       |
| 退出路徑 | 使用者如何離開：返回、導航他處、取消、完成後自動轉場       | 最常遺漏 — 設計注意力集中在「進來之後」     |

「顯示」和「可用操作」的區別在互動性：顯示一顆重試按鈕是「顯示」、按下重試按鈕觸發重連是「可用操作」。

## 填寫範例

以一個遠端終端機 app 的連線畫面為例：使用者從首頁點入、畫面自動建立 WebSocket 連線，狀態機有 idle / connecting / connected / error / disconnected 五個狀態。實作審查時填出的矩陣：

| 狀態         | 顯示                   | 可用操作     | 進入條件      | 退出路徑 |
| ------------ | ---------------------- | ------------ | ------------- | -------- |
| idle         | 空白（自動連線）       | 無           | 進入畫面      | 無       |
| connecting   | 進度指示               | 無           | idle 自動觸發 | 無       |
| connected    | 終端機 + 工具列        | 打字、特殊鍵 | 連線成功      | 無       |
| error        | 錯誤訊息 + 重連按鈕    | 重新連線     | 連線失敗      | 無       |
| disconnected | 「連線中斷」+ 重連按鈕 | 重新連線     | 連線斷開      | 無       |

退出路徑欄全空 — 五個狀態、五個死胡同。每個狀態單獨看功能都正常（error 有重連、connected 能打字），只有整張表攤開才看得到結構性缺口。修復後每個狀態補上至少一條退出路徑：

| 狀態         | 顯示                   | 可用操作               | 進入條件      | 退出路徑                 |
| ------------ | ---------------------- | ---------------------- | ------------- | ------------------------ |
| idle         | 空白（自動連線）       | 取消                   | 進入畫面      | 取消 → 返回首頁          |
| connecting   | 進度指示 + back        | 取消連線               | idle 自動觸發 | back → 返回首頁          |
| connected    | 終端機 + 工具列 + back | 打字、特殊鍵、中斷連線 | 連線成功      | back / disconnect → 首頁 |
| error        | 錯誤訊息 + 重連 + back | 重新連線、返回         | 連線失敗      | back → 返回首頁          |
| disconnected | 中斷訊息 + 重連 + back | 重新連線、返回         | 連線斷開      | back → 返回首頁          |

## 填寫步驟

1. **列出畫面的所有狀態**。從狀態管理機制取得（enum、state 變數、reactive data）；沒有明確 enum 時從視覺變化反推 — 每種不同的視覺呈現就是一個狀態。
2. **每個狀態填四欄**。填不出來的欄位先留空 — 留空本身就是發現。
3. **檢查退出路徑欄**。空格 = 死胡同，需要補退出路徑。
4. **檢查可用操作欄**。操作欄為空可能合理（等待中），也可能是缺互動設計 — loading 狀態通常應有「取消」、error 狀態通常應有「重試」和「返回」。

一個 5 狀態的畫面約 10 分鐘填完，能同時暴露全部導航缺口 — 遠低於實機測試才發現的成本。

## 從 BDD 操作盤點展開

BDD 操作盤點（Given / When / Then）聚焦「什麼情境下做什麼得到什麼」，Then 通常描述可觀察結果 — 「畫面顯示 X」比「使用者可以做 Y」更容易寫成斷言，因此盤點系統性偏向描述顯示、遺漏操作與退出。展開步驟：

1. 從盤點列出所有畫面（每個情境至少涉及一個畫面、去重）
2. 每個畫面列出狀態（從 Given / Then 條件抽：「Given 連線已建立」暗示 connected 狀態；同時對照程式碼 enum — 盤點沒提到的狀態同樣需要設計）
3. 每個狀態填「顯示」欄（盤點的強項、直接搬）
4. 每個狀態補「可用操作」和「退出路徑」欄 — 盤點通常不提供，對每個狀態問兩個問題：使用者在這個狀態下想做什麼？怎麼離開？
5. 檢查空白格

## 路由可達性檢查

矩陣管畫面內部，路由可達性管畫面之間：比較 router 定義的所有路由、和使用者從 UI 操作能到達的所有路由，兩個集合的差集就是問題。

- **router 有但 UI 不可達**：路由定義了、畫面實作了，但沒有任何 UI 元素導航過去 — 功能存在但使用者找不到入口。這是「寫了 function 沒人呼叫」的 UX 版死程式碼。
- **UI 指向但 router 沒有**：按鈕導向未定義的路由 — 404 或空白畫面。

檢查方法依規模選擇：路由少（< 10）逐一在 UI 上找到通往該路由的操作路徑；路由多時從操作盤點交叉比對（每個使用者操作的入口畫面是否從首頁可達），或自動化 — 解析 router 定義的路徑集合、grep UI 程式碼中所有導航呼叫的目標路徑、取差集。自動化能抓靜態缺口，動態導航（執行期條件決定目標）仍需人工確認。

交叉比對時注意操作的執行端：只有「使用者在這個 app 端執行」的操作需要 UI 入口，主機端 / 後台操作不需要。

## 容易漏列的狀態：initializing

畫面查詢的對象有自己的生命週期時（browser extension 的 service worker、遠端服務、另一個 process 的常駐程式），畫面開啟瞬間「對方還沒醒、還不知道狀態」是常態 — initializing 是真實狀態，在矩陣裡需要自己的一行：顯示「初始化中」、退出路徑是收到回應轉入正常態、timeout 後才轉入離線或錯誤。

把 initializing 與離線合併的畫面，每次冷啟動都閃現一段假離線 — 使用者據此做錯誤決策（放棄、重裝、回報故障）。對策：被查詢方在初始化期間就能回應 baseline 的 initializing 狀態（「我在、還沒準備好」與「沒有回應」是不同資訊）；過渡狀態的顯示必須有資料來源與退出條件，硬編的「檢測中...」永遠不會離開、等於把過渡態寫成死胡同。列狀態時多問一句：查詢對象跟畫面同生命週期嗎？不同就補 initializing。

## 反模式：假設使用者只走 happy path

UI 設計只覆蓋「一切順利」的情境 — 連線成功後打字、配對成功後連線 — 忽略非順利情境的需求：等待中想取消、失敗後想換方向、成功後想離開、中途想做其他事。

隱性假設的形態：「connected 後使用者不會想回首頁」「使用者遇到錯誤會想重試」（沒考慮想放棄）。這些假設在企劃、設計、實作每一層各自累積，最終產出的 app 在正常情境運作良好、任何偏離情境使用者都被困住。

矩陣之上的三條加強檢查：

- **退出路徑欄為空** → 補退出路徑。即使是過渡狀態（connecting）使用者也應該能取消。
- **可用操作欄只有一個選項** → 這條路走不通時使用者被困。至少考慮一條替代路徑（返回 / 取消）。
- **進入條件不可逆** → 使用者無法自行觸發反向操作（「連線成功」的反向「斷開連線」不存在）代表進入後只能等系統狀態變化。

## 退出路徑的例外邊界

退出路徑是預設要求。可豁免的唯一情況：該狀態持續時間極短（< 1 秒）且有保證的自動轉場 — 即使如此仍建議保留取消，因為「極短」是正常情況的預期、異常情況（斷網、服務當機）會讓這個狀態停留很久。iOS HIG 和 Material Design 對 modal 畫面都要求 dismiss 機制，退出路徑欄是這個要求的普遍化檢查。

## 檢查清單

- [ ] 每個畫面的每個狀態都在矩陣裡有一行？
- [ ] 查詢對象有獨立生命週期時，initializing 是獨立狀態（與離線 / 錯誤分開、有 timeout 退出）？
- [ ] 每個狀態至少一條使用者可自行觸發的退出路徑？
- [ ] 過渡狀態（loading / connecting）提供取消？
- [ ] error 類狀態同時有「重試」和「返回」？
- [ ] 可用操作欄只有一個選項的狀態，確認過那條路走不通時的後果？
- [ ] router 定義的每條路由都有 UI 入口（或明確標記為 deep link / 程式內部專用）？
- [ ] UI 的每個導航目標都在 router 有定義？
- [ ] 矩陣已轉成（或規劃轉成）UI test case？
