# Gate 與 Fallback：關卡失敗時使用者的路

本 reference 為「設計或審查使用者必須通過的關卡」情境。

適用：生物辨識 / 密碼認證、網路連線檢查、系統權限請求（相機 / 位置 / 通知）、硬體或版本條件檢查；審查「條件不滿足時使用者被擋死」的缺口；規劃實機測試時判斷哪些關卡在開發環境測不到。
不適用：畫面內部的狀態轉換設計（屬狀態矩陣範疇）；錯誤發生後的恢復流程（屬錯誤恢復範疇）— gate 管「進得去嗎」，錯誤恢復管「進去之後出錯怎麼辦」。

> **自包含聲明**：閱讀本文件不需要先讀其他 reference。

---

## 核心概念

Gate 是使用者操作流程中「必須通過才能繼續」的關卡。Gate 設計的核心責任是確保使用者在每種結果下都有路可走，而非只設計「通過」的情境。Fallback 是 gate 未通過時的替代路徑：降級功能、替代驗證方式、手動重試、或放棄返回。

## 每道 gate 的必答問題

設計每道 gate 時回答三個問題（成功 / 失敗 / 不確定），任何一題答不出來就是設計缺口：

**成功時做什麼** — 通過後進入下一步。最直覺、最不容易遺漏。

**失敗時做什麼** — 未通過時的替代路徑。最容易遺漏的一題。替代路徑的選項：降級功能（部分可用）、替代驗證（密碼代替 Face ID）、手動重試、放棄操作（返回上一頁）。實例：一個自用工具 app 的 biometric gate 設了「只接受生物辨識」，Face ID 不可用時使用者直接被擋住 — 沒有密碼 fallback、沒有跳過、沒有返回；修復只是改一個參數讓系統 fallback 到裝置密碼，但這個決策本該在企劃階段做、而非實機測試時才發現。

**使用者不知道發生什麼時做什麼** — gate 處理中（loading）或結果不確定（timeout）時的呈現：認證彈窗因系統延遲未出現、請求已發未回應、權限對話框被其他 dialog 遮擋。這個狀態下使用者需要：知道系統在做什麼（loading 指示）、可以取消等待、超過合理時間有提示（timeout 訊息 + 重試選項）。

把三題答案整理成 gate 設計表、作為規格的一部分：

| Gate     | 成功         | 失敗                | 不確定              |
| -------- | ------------ | ------------------- | ------------------- |
| 生物辨識 | 進入主畫面   | 提示輸入裝置密碼    | 顯示「驗證中」      |
| 網路連線 | 開始載入資料 | 顯示離線提示 + 重試 | 顯示 loading + 取消 |
| 相機權限 | 開啟掃描功能 | 說明原因 + 設定連結 | 等待系統對話框      |
| 藍牙     | 開始裝置搜尋 | 提示開啟藍牙 + 連結 | 顯示搜尋中 + 取消   |

失敗欄和不確定欄為空的 gate 就是 UX 死胡同候選。

## Gate 類型與各自的設計重點

**認證 gate**（生物辨識、密碼、PIN、OAuth）：fallback 設計取決於安全需求 — 銀行 app 可能要求雙重驗證、沒有更低層級 fallback；自用工具的使用者就是 owner、可用性優先於認證強度。取捨要顯式記錄（見下方記錄格式）。

**網路 gate**：狀態是連續的（連線中 / 已連線 / 斷線 / 不穩定），比二元結果的 gate 難處理。三種策略依功能對即時連線的依賴程度選擇：

- **Offline-first**：核心操作在本地完成、網路只負責同步。前提是資料可本地存儲且衝突可解決（筆記、待辦）。不需要 gate — 網路狀態不阻擋操作，UX 重點是同步狀態可見（已同步 / 待同步 / 失敗）。
- **Retry with feedback**：功能需要網路但可等待（即時連線類）。斷線時顯示狀態 + 手動重試 + 退出路徑，自動重試要有上限和間隔遞增。
- **Degraded mode**：部分功能依賴網路。斷線自動降級、清楚標示哪些功能不可用，不阻擋其餘操作。

連線存在但品質差（高延遲、頻繁斷開）比完全離線更難：每次請求可能成功可能逾時。處理：合理的逾時時間、逾時後顯示狀態與重試選項（不自動重試、避免在不穩定網路上累積請求）、loading 狀態提供取消。

**權限 gate**（相機、位置、通知、麥克風）：特殊性在使用者可以永久拒絕 — 拒絕後再次請求不會彈系統對話框，唯一路徑是引導到系統設定手動開啟。第一次請求的時機和說明直接影響授權率：

- 時機：多數場景推薦「功能使用時即時請求」（使用者在操作 context 中、理解為什麼需要）而非首次開啟一次性請求（使用者尚未使用功能、傾向拒絕）。首次開啟請求只適合權限少（1-2 個）且與核心功能直接相關（相機 app 請求相機）。
- Pre-permission 說明畫面：觸發系統對話框前先用自己的畫面解釋用途 —「需要相機來掃描 QR code」而非 API 名稱；提供「稍後再說」；可視覺化展示授權後的功能。
- 拒絕後：記住拒絕狀態（不反覆詢問）、提供功能降級（掃碼改手動輸入）、適當時機用非侵入提示再說明、提供直接跳轉 app 設定頁的按鈕。

**環境 gate**（OS 版本、特定感測器、藍牙開啟）：fallback 通常有限 — 硬體不存在無法軟體模擬，但至少告知為什麼功能不可用、而非靜默禁用。

**破壞性操作確認 gate**：操作的破壞半徑大於使用者直覺預期時（不可逆、覆蓋或清空既有資料 —「匯入」聽起來是加法、語意可能是取代，落差越大越需要確認），執行前攔一道描述具體後果的確認（「將清空現有 N 筆」、不只問「確定嗎」）。這類 gate 的特化要求是**確認機制本身的故障模式也要設計**：確認元件沒渲染、事件沒綁上、回應逾時，一律視為未確認、不執行破壞 — 安全預設倒向不可逆性低的那邊（不清空可以重試、清空無法還原）。程式碼層檢查訊號：確認邏輯的 else / null 分支走向哪邊。

另外兩類實務常見：**付費 gate**（失敗路徑的目標是引導付費而非替代功能、取捨依商業模式）、**版本相容 gate**（提示更新，注意強制更新會擋住無法更新的舊裝置使用者）。

## 安全 vs 可用性的顯式記錄

Gate 嚴格度的決策（例如「只接受生物辨識、不允許密碼 fallback」）涉及安全和可用性的取捨，應在功能規格中顯式記錄，讓 code review 和後續維護者理解決策背景：

```text
Gate: biometric authentication
Decision: allow device credential fallback
Security trade-off: device credential (PIN/password) is weaker than biometric
Rationale: self-hosted tool, user = owner, availability > auth strength
Risk accepted: someone with device PIN can access the app
```

沒有記錄的嚴格設定通常不是決策、是預設值或直覺 —「安全性更高」的原始意圖沒有對照過「條件不滿足時使用者完全無法使用」的代價。

## 生物辨識的失敗情境與平台行為

失敗有四類，各自合理的 fallback 不同：

| 失敗類型     | 例子                             | 合理 fallback      |
| ------------ | -------------------------------- | ------------------ |
| 暫時性       | 光線不足、手指潮濕               | 重試可能成功       |
| 持續性       | 戴口罩、手指受傷                 | 需要替代認證       |
| 硬體不可用   | 無生物辨識模組、模擬器、MDM 禁用 | 需要替代認證       |
| 使用者未設定 | 有硬體但沒設 Face ID / 指紋      | 引導設定或替代認證 |

平台 API 行為（以 Flutter `local_auth` 的 `biometricOnly` 參數對應）：

- **iOS（LocalAuthentication）**：`deviceOwnerAuthenticationWithBiometrics` 只接受生物辨識；`deviceOwnerAuthentication` 失敗後系統自動彈出裝置密碼。認證 UI 由系統控制、多次失敗會自動鎖定。
- **Android（BiometricPrompt）**：`BIOMETRIC_STRONG` / `BIOMETRIC_WEAK` / `DEVICE_CREDENTIAL` 三個等級可用 `|` 組合，`BIOMETRIC_STRONG | DEVICE_CREDENTIAL` 即「先生物辨識、失敗 fallback 到裝置密碼」。廠商間辨識品質差異大。

## 開發環境 vs 真機的行為差異

開發環境的 gate 行為比真機寬鬆，讓缺口在開發階段不可見 — 和 mock 遮蔽 protocol 問題同構。三個機制：模擬器不支援硬體功能（生物辨識、藍牙、push）導致 app 跳過認證走預設路徑；debug build 的權限行為可能不同；test 環境 mock 掉所有 gate（fake service 永遠成功）。

對策是在規格中建一張差異表，「模擬器行為」和「真機 release」不同的行 = 必須上真機驗證的項目：

| Gate         | 模擬器行為            | 真機 release    | 風險                            |
| ------------ | --------------------- | --------------- | ------------------------------- |
| 生物辨識     | 跳過（硬體不可用）    | 正常            | 模擬器上看不到 fallback 缺失    |
| 網路連線     | 通常正常（host 網路） | 行動網路 + WiFi | 模擬器網路狀態不代表行動網路    |
| 相機權限     | 無相機 / 虛擬相機     | 正常            | 無法測試真實權限流程            |
| 藍牙         | 不支援                | 正常            | 藍牙相關功能完全跳過            |
| Push 通知    | 不支援（iOS 模擬器）  | 正常            | 通知觸發的導航路徑不可測        |
| App 簽名驗證 | debug 簽名自動通過    | release 簽名    | 簽名相關 gate 只在 release 生效 |

用法：開發時對照哪些 gate 尚未被真實驗證；實機測試計畫對每一行設計案例（涵蓋失敗路徑）；code review 看到嚴格 gate 設定時提示「模擬器上看不到這個問題」。

## 檢查清單

- [ ] 每道 gate 都回答了成功 / 失敗 / 不確定必答問題？
- [ ] 失敗路徑至少有一個選項（替代驗證 / 降級 / 重試 / 返回）？
- [ ] 不確定狀態有 loading 指示 + 取消 + timeout 提示？
- [ ] Gate 嚴格度的取捨有顯式記錄（決策、代價、接受的風險）？
- [ ] 權限請求在功能使用時觸發、有 pre-permission 說明？
- [ ] 權限被永久拒絕後有引導到系統設定的路徑？
- [ ] 網路類功能選定了策略（offline-first / retry / degraded）並涵蓋「連線但品質差」？
- [ ] 覆蓋 / 清空 / 取代既有資料的操作有後果描述的確認、且確認機制故障時預設不執行？
- [ ] Gate 行為差異表列出了哪些項目必須上真機驗證？
