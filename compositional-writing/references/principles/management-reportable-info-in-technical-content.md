# 技術教材要內嵌管理層可彙報的資訊

> **角色**：本卡是 `compositional-writing` 的支撐型原則（principle）、被 writing-articles reference 引用。
>
> **何時讀**：寫技術教材時，判斷段落是否需要補充成本、時程或進度資訊。

## 原則

技術教材的讀者通常需要向上彙報：為什麼要做這件事、要花多久、花多少錢、做完怎麼知道有效果。如果教材只講技術機制，讀者學會了做法卻沒有素材向管理層說明，結果是做法正確但推不動。

管理層需要的資訊有四類，跟技術段落的嵌入位置明確對應：

| 資訊類型   | 管理層的問題                 | 嵌入位置                         |
| ---------- | ---------------------------- | -------------------------------- |
| 成本量級   | 「這個選項每月多花多少？」   | 緊接取捨討論段，用量級而非精確數 |
| 時程估算   | 「導入/遷移/拆分要多久？」   | 緊接操作流程段，給範圍不給單點   |
| 進度指標   | 「怎麼知道有在進步？」       | 治理或 review 流程段             |
| 決策簽核點 | 「這個決定要不要上層同意？」 | 影響全組織的架構決策段           |

## 嵌入原則

- 嵌在技術段落旁邊、1-2 句帶過量級，不獨立成段
- 成本用量級而非精確數字（「月費約 $32」而非「$31.54」）
- 時程用範圍而非單點（「1-2 週」而非「10 天」）
- 進度用可查詢的指標（「缺 tag 資源數 / 總資源數」）
- 成本數字標明是量級參考，雲端定價會變但量級通常穩定

## 判讀訊號

對每個取捨段落問：「讀者的老闆會問什麼？」如果答案是「多少錢」或「多久」而段落裡沒有，就是缺口。影響跨團隊或跨帳號的決策要標記為需簽核。
