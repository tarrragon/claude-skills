# Source to Teaching Analysis

本 reference 為「把外部分析文章或高密度研究材料，轉成教學型分析文章」情境。適用於分析師文章、投資人備忘錄、產業評論、研究摘要、AI 轉寫稿，也適用於從業者經驗談（訪談、社群貼文、口述整理）。目標是把 source 轉成讀者可重用的判讀框架，不是只做摘要、翻譯或語氣改寫。

> **自包含聲明**：本文件不依賴其他 reference。讀完本文件即可獨立執行外部分析材料到教學文章的轉換。

---

## 何時參閱本文件

| 觸發情境                                            | 動作                                        |
| --------------------------------------------------- | ------------------------------------------- |
| 要把外部分析師文章改寫成自己的教學文章              | 跑完整 source-to-teaching pass              |
| 來源是從業者經驗談 / 訪談 / 口述整理 / 社群實戰貼文 | 先跑 source type gate、走機制重建 pass      |
| 改寫稿結論都在、但斷言靠口吻與權威支撐              | 跑機制重建 pass                             |
| AI 改寫稿讀起來流暢、但像原文摘要                   | 跑 deliverable pass                         |
| 文章術語密度像寫給同業、目標讀者跟不上              | 跑 reader-level pass                        |
| 來源文章的觀點與本文結論混在一起                    | 跑 source layering pass                     |
| 文章已套決策框架、但正文像分析過程記錄              | 跑 title promise 與 teaching narrative pass |

不適用：純翻譯逐句校稿、學術論文正式摘要、內部 analyst-to-analyst 報告。這些產物可以保留原文的 reader contract 或 process metadata。

---

## 為什麼外部分析改寫需要獨立指引

外部分析文章同時帶有三種東西：事實材料、原作者 frame、原作者 reader contract。AI 或人類若只做風格轉換，會把三者一起搬進新文章。結果是文字符合目標風格，但結論仍沿用原作者視角，讀者仍需要原領域背景，文章也缺少可遷移的教學框架。

教學型分析文章要做的是「source transformation」：保留可驗證事實，辨識原作者判讀，重建適合目標讀者的推導，最後交付可重用的判斷問題與預警訊號。

---

## Source Type Gate：分析文還是經驗談

轉換流程的第一步是判別 source 自帶什麼。兩類 source 的轉換工作本質不同，混用流程會產出錯的知識類型。

| Source 類型                                  | 自帶的層                        | 轉換的核心工作                                     |
| -------------------------------------------- | ------------------------------- | -------------------------------------------------- |
| 分析文（分析師文章、備忘錄、產業評論）       | 事實 + 原作者判讀 + 分析層      | **拆層**：分開三層、防止原作者 frame 被當成事實    |
| 經驗談（訪談、社群貼文、口述整理、實戰分享） | 事實 + 從業者判讀、**無分析層** | **重建**：補建機制，把判讀從結論升級成可推導的判準 |

判別訊號：source 的斷言靠什麼支撐。分析文的斷言後面有數據、機制、市場結構；經驗談的斷言後面是經驗本身（「做這行都知道」「資深的人都這樣做」）。經驗談的斷言是從業者走完推導後輸出的最後一步——推導路徑留在從業者腦中，素材裡只有結論。

分析文走下方五個 Pass。經驗談先走機制重建 pass，再接 Pass 2（reader level）與 Pass 5（deliverable）。

---

## 經驗談 Source：機制重建 Pass

**經驗談轉教學的核心工作是重建分析層：每個承擔判準責任的斷言，都要能回答「為什麼成立、在什麼條件下不成立」。** 只做系統化整理——分主題、換書面語、加表格、補術語卡——會產出「披著教學結構的經驗談」：結構合規、結論都在、支撐全缺，讀者拿到的是要相信的規則，不是能自己推的判斷。

三層分工：

| source 內容層                          | 處理方式                                    |
| -------------------------------------- | ------------------------------------------- |
| 事實（規則條款、流程、事件）           | 保留，構成教材的 fact 層                    |
| 從業者判讀（閾值、口訣、「該這樣做」） | 當 hypothesis，重建支撐機制後才升為教材判準 |
| 分析層（機制、量化、結構解釋）         | source 沒有自帶，由作者從領域通用知識補建   |

機制重建的操作，對每個承擔判準的斷言問四個問題：

1. **這個斷言背後的成本結構是什麼？** 例：「絕不允許 X 發生」通常意味著 X 的代價跟預防成本嚴重不對稱——把兩側成本各自展開成可估算的組成，不對稱從口號變成估算框架。
2. **這個閾值是什麼的反推值？** 例：「不超過七成」「至少養兩家」這類 magic number 是某個約束條件的反推——找出約束（替補產能上限、驗證週期），數字降級為機制的例示，讀者換情境能重算。
3. **這個規則的設計者圖什麼？** 例：某個交易規則存在，是誰設的、保護誰的什麼利益——補上設計者的誘因分析，規則從「要背的條文」變成「可預測的行為」。
4. **這個操作對應什麼既有分析語言？** 例：「可無條件取消就先下單卡位」是免費選擇權、「規則不同下單膽量不同」是風險敞口管理——套上領域通用的分析語言後，多個操作能從同一機制推導出來。

重建的知識來源有兩條：領域通用理論（經濟學、賽局、選擇權思維、通路結構），或回頭追問原始從業者「為什麼是這個數」。素材與作者理論掌握度決定走哪條；兩條都走不通的斷言，標明「從業者經驗值、機制待補」，別偽裝成已推導。

**重建有停止線**：只有承擔判準責任的斷言需要機制。氛圍性敘述刪除或降為 hook；心態與職場修為內容是合法的經驗分享，但知識類型是 career essay——分開成篇並標明定位，別混在分析教學裡稀釋兩邊。

反例（只換口吻）：

```markdown
量產初期放量時，不會讓單一家就超過七成，先分散再看各家表現逐步調整。
```

問題：七成是從業者的壓縮結論，教材原樣轉述，讀者換情境無法重算。

正例（重建機制）：

```markdown
量產初期的放量上限，由「單一家爬坡失敗時、其餘供應商短期能吸收的產能」反推。
替補家能在一個交期內補上三成，主力家的上限就是七成——比例跟著替補產能彈性變動，
替補弱就要壓低主力佔比、多養一家。
```

## 五個 Pass

### Pass 1：Source Layering

Source layering 是把材料拆成事實、原作者判讀、本文推導。每句材料進正文前先問：

| 問題                                 | 層級       |
| ------------------------------------ | ---------- |
| 這件事有公開來源可驗證嗎？           | 事實       |
| 這是不是原作者或市場敘事的解釋？     | 原作者判讀 |
| 這是不是本文從多個材料合成出的框架？ | 本文推導   |

寫法規則：

- 事實放在背景或事件段。
- 原作者判讀只當 hypothesis prior，不直接當本文結論。
- 本文推導要明確標成本文整理出的框架或判斷問題。

反例：

```markdown
這代表基礎設施廠商正在垂直整合 data pipeline。
```

問題：若「垂直整合」是本文推導，這句把推導寫成事實。

正例：

```markdown
本文把這個收購視為「基礎設施廠商往 data pipeline 延伸」的訊號。這是本文判讀，不是事件本身。
```

### Pass 2：Reader Level

Reader-level pass 是把原文 reader contract 改成目標讀者 contract。先辨識原文寫給誰，再決定是否降一級。

| 原文類型     | 常見 reader contract   | 轉換策略                 |
| ------------ | ---------------------- | ------------------------ |
| 大眾新聞     | 只需要知道事件         | 補機制與可遷移框架       |
| 投資人備忘錄 | 預設懂市場 shorthand   | 降術語密度、補因果鏈     |
| 產業內部文   | 預設懂供應鏈與競爭格局 | 補背景角色與資料形狀     |
| 教材型文章   | 已經有教學推導         | 檢查是否符合本站段落責任 |

檢查法：

- 三句內連續三個以上跨領域術語，拆段或補卡。
- 一句話跨三個以上因果步，拆成多段。
- 讀者能背結論但說不出推導，補機制、公式、數字或例子。

### Pass 3：Internal Analysis

Internal analysis 是作者背後的 hypothesis 探索，不等於文章章節。可以用 WRAP 或其他決策框架做內部檢查：

- 要回答什麼問題？
- 手上 evidence 夠不夠？
- 有哪些本質不同的合理解釋？
- 每個解釋的 evidence weight 是什麼？
- 哪些觀察會推翻本文判讀？

這些問題要做完，但不直接當正文標題。正文標題描述讀者會學到什麼，不描述作者正在執行哪個分析步驟。

### Pass 4：Title Promise

Title promise 是標題對讀者做的合約。跑完內部分析後，列出文章每段是否服務標題承諾。

| 檢查項                                 | 行動                           |
| -------------------------------------- | ------------------------------ |
| 某段超過 20% 篇幅但不在標題承諾內      | 壓縮成背景句、移到別篇、或刪除 |
| 來源 prior 比正文機制更長              | 縮減 prior，讓主體回到本文承諾 |
| 標題承諾的是框架、正文卻在解釋作者動機 | 改寫主體或重命名標題           |

Title promise pass 保護讀者不被作者的分析過程帶離主題。

### Pass 5：Deliverable

Deliverable pass 確認文章交付的是可遷移框架，而不是原文摘要。結尾至少回答：

- 下次遇到同類事件要看哪些訊號？
- 哪些機制若成立，本文判讀才成立？
- 哪些觀察會削弱或推翻本文判讀？
- 讀者下一步該看哪個概念、卡片或案例？

常見責任順序：

```text
事件本身 → 結構性機制 → 長期影響 → 預警訊號 → 可遷移框架
```

這不是固定模板，而是責任清單。不同文章可以調整順序，但不能缺少可遷移框架與失效條件。

## 自檢清單

- [ ] 已判別 source 類型（分析文走拆層、經驗談走機制重建）。
- [ ] 經驗談 source：承擔判準的斷言都有機制支撐，訴諸權威（「資深的人都這樣」）沒有被沿用為教材論據。
- [ ] 經驗談 source：數字閾值都找到了反推邏輯、或標明「經驗值、機制待補」。
- [ ] 每個來源句子都能標成事實、原作者判讀、本文推導之一。
- [ ] 原作者 frame 沒有被寫成事件事實。
- [ ] 目標讀者不需要原文 reader contract 也能理解主體推導。
- [ ] 三句內沒有未解釋的連續跨領域術語。
- [ ] 內部分析框架沒有外露成正文標題。
- [ ] 正文段落都服務 title promise。
- [ ] 結尾提供可遷移框架、預警訊號與下一步路由。

## 反模式速查

| 反模式                           | 修法                                      |
| -------------------------------- | ----------------------------------------- |
| 只要求 AI「改成我們的風格」      | 改成「抽出可遷移判讀框架」                |
| 經驗談只做主題分類 + 書面化      | 跑機制重建 pass、把判讀升級成可推導的判準 |
| 教材沿用「厲害的人都這樣」當論據 | 重建該斷言背後的機制、權威降為佐證        |
| 同一句 anecdote 攤進多篇         | 找出它支撐的機制、收斂到機制的歸屬篇章    |
| 把原作者分類直接當本文標題       | 標出這是原作者 frame 或本文 frame         |
| 引用機構名但沒有具體來源         | 刪具名 attribution 或重新查證             |
| 章節標題是分析 process           | 改成教學標題                              |
| 文章結尾只有事件結論             | 補下次遇到同類事件的判斷問題              |
| 讀者需要懂原領域 shorthand 才懂  | 降術語密度，補 bridge sentence            |

**核心原則**：外部分析文章改寫的核心是 source transformation。保留事實、分離原作者判讀、重建目標讀者可理解的推導，最後交付可遷移框架。
