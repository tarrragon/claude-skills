# 讀程式自己的 log，從症狀往上游找

> **角色**：本卡是 `linux-install-debug` 的支撐型原則（principle）、被 SKILL.md 與 [read-logs](../read-logs.md) 情境引用。
>
> **何時讀**：症狀是「某個程式行為不對」而終端機輸出不足以定位根因時。

當現象是某程式行為不對，它自己的 log 幾乎總是比終端機捲過的畫面更接近真相。很多程式在終端機只印一段摘要，卻同時把詳細執行紀錄寫進 log 檔或系統日誌；終端機訊息不足時，那份 log 裡往往就有明確答案。除錯的一個核心習慣是：找程式自己的 log，而不是只盯著終端機捲動。

## 為什麼終端機不夠

- 終端機輸出易逝、會被後續沖掉、多來源交錯。
- 很多失敗的「表面錯誤」離「真正原因」隔好幾層：一個指令因為前面某個檔案不存在而失敗，終端機可能只報一個看似無關的下游錯，但 log 裡有那句 `File does not exist` 直指源頭。
- 畫面可能完全沉默（什麼錯都沒印），但 log 有記 —— 例如某 shell 換配色沒生效、畫面無異狀，log 一句「讀取 scheme 檔失敗：檔案不存在」點出根因。

## 找 log 的去處

- 程式自己的 log 檔：常在 `~/.local/state/<程式>/`、`~/.cache/<程式>/`、`/var/log/`。
- systemd 服務：`journalctl -u <unit>`（`-b` 本次開機、`-f` 跟隨、`--since`）。
- 程式啟動時印出的 log 路徑（啟動訊息常有 "logging to ..."）。
- 二進位格式的 log（如某些 shell 的自訂 log）：用 `strings <file>` 過濾出可讀文字再 `grep`。

## 讀法：用症狀當關鍵字往上游搜

拿到 log 後，用症狀 / 錯誤字串當關鍵字往上游搜，找「第一個」異常，而不是停在最後一個下游錯。時間戳、行號、`ERROR` / `failed` / `not found` / `does not exist` 這些是好的搜尋錨點。找到源頭那一行，通常就是原本會猜錯的地方。

這跟「產生可診斷的 log」是一體兩面：寫自己的腳本時，一開始就 `tee` 全部輸出進帶時間戳的 log、加 `ERR` trap 印出出錯行與指令，未來除錯就從瞎找變成定位。
