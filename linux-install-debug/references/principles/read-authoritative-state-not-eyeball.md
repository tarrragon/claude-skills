# 讀權威狀態，不靠肉眼猜表象

> **角色**：本卡是 `linux-install-debug` 的支撐型原則（principle）、被 SKILL.md 的「最高紀律」與四步流程引用。
>
> **何時讀**：任何要判斷「某個東西現在是什麼狀態」的時候 — 鎖了沒、活著沒、服務歸誰、通不通、滿了沒。

診斷一個問題時，第一個動作不是猜「這看起來像什麼」，而是問「這件事的權威狀態記在哪、我怎麼讀它」。畫面現象、終端機殘影、視窗長相都是表象；能定案的是系統裡記錄這件事的權威來源。把判斷建立在權威來源上，是快速且不猜錯的除錯基礎。

## 表象會騙人的三種形態

- **畫面誤導**：畫面上有某個元素，不等於背後的狀態成立。密碼框 ≠ session 被鎖（可能只是內嵌鎖屏樣式 widget 的儀表板）；通知沒跳 ≠ 沒送出（可能勿擾吃掉）；進度條卡住 ≠ 掛了。
- **讀錯層**：權威來源分層，查錯層會得到誤導的「否定」答案。Wayland 合成器層的 `ext-session-lock` 用 logind 的 `LockedHint` 查 — 天生查不到，不是沒鎖，是查錯層。**你寫的驗證腳本本身也是一隻會讀錯層的眼睛**：檢查一個 config 有沒有被 GNU stow 部署時，若判「這個 leaf 檔是不是 symlink」會假陰性 — stow 摺疊目錄時 symlink 在父目錄那一層、leaf 是穿透摺疊走到 repo 的實檔；權威讀法是 `test <leaf> -ef <repo檔>`（比對 inode、跟隨符號連結）或 `ls -ld` 逐層看，不是對單檔 `-L`。
- **查詢條件錯**：用對了工具、問錯了條件。用一個「你以為的」process 名 `pgrep`，程式其實以 symlink 名 / 別名在跑，掃不到不代表它死了。

## 為什麼容易猜錯兩次

第一次猜錯來自畫面（看到就以為懂了）。第二次猜錯往往來自「看似嚴謹的檢查」，但那些檢查讀的是錯的層或錯的條件，反而給出一個更有說服力的錯誤結論。真正的解法不是再加一個同類檢查，是換去讀**這件事真正的權威來源**（那個程式的 log、那個協議的狀態）。連續兩個檢查都在錯的層打轉，加再多也跨不過去。

## 操作紀律

1. 症狀先描述、不下結論。
2. 對照「症狀類別 → 權威來源 → 工具」（見 SKILL.md 速查表）找到權威來源。
3. 確認你讀的層跟查詢條件都對（是這件事的權威來源嗎？comm 名 / service 名 / session id 對嗎？）。
4. 權威狀態跟肉眼第一印象矛盾時，信權威、回頭修正第一印象。

**遠端除錯反而逼出這套紀律**：SSH 看不到畫面，就只能讀權威狀態，自動避開「看畫面下結論」。把這套「沒有畫面、只信權威」的紀律也用在本地。
